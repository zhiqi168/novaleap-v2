# NovaLeap 项目架构详解

> 本文档基于 `nova-backend` `nova-frontend` `nova-admin` 三个子系统的源码完整分析整理，覆盖后端、前端、安全、AI、数据库、部署全链路设计。
> 最后更新：2026-07-18

---

## 一、项目概述

NovaLeap 是一个面向学习、内容沉淀与社区互动的全栈项目，采用前后端分离架构。

### 1.1 子系统

| 子系统                    | 技术栈                                                                    | 说明                    |
| ---------------------- | ---------------------------------------------------------------------- | --------------------- |
| **nova-frontend**（用户端） | Vue 3.4 + Vite 5 + Pinia 2 + Tailwind CSS 3                            | 面向普通用户的主站，所有路由懒加载     |
| **nova-admin**（管理端）    | Vue 3.4 + Vite 5 + Pinia 2 + Tailwind CSS 3                            | 面向运营/管理员的后台，无懒加载      |
| **nova-backend**（后端）   | Spring Boot 3.4 + MyBatis-Plus 3.5 + MySQL 8 + Redis 7 + Spring AI 1.0 | 统一 API 服务，同时服务用户端和管理端 |

### 1.2 部署架构

Docker Compose 一键部署，5 个容器服务：

| 服务              | 镜像                            | 默认端口 | 数据持久化                  |
| --------------- | ----------------------------- | ---- | ---------------------- |
| `nova-mysql`    | MySQL 8.0                     | 3306 | volumes 挂载 mysql-data/ |
| `nova-redis`    | Redis 7.2 Alpine              | 6379 | volumes 挂载 redis-data/ |
| `nova-backend`  | eclipse-temurin:17-jre-alpine | 8080 | 无状态                    |
| `nova-frontend` | nginx:alpine + Vue 构建产物       | 80   | 无状态                    |
| `nova-admin`    | nginx:alpine + Vue 构建产物       | 8081 | 无状态                    |

服务器与域名均采购自阿里云。因国内备案流程要求，服务器部署在海外节点，前端 Nginx 容器内配置 `proxy_pass` 指向同一 Docker 网络内的后端容器，避免跨域问题。

## 二、项目亮点总结

### 2.1 统一后端 + 双 Portal 权限隔离

用户端和管理端共用同一套后端服务，不拆分为两个独立后端应用。通过 Spring Security 的 `PORTAL_CLIENT` / `PORTAL_ADMIN` 权限体系 + `WebExpressionAuthorizationManager` 组合表达式，实现物理级的路径安全隔离。用户端 Token 即使泄露也无法访问任何管理端接口。

### 2.2 七层邮箱验证码防刷体系

| 防护层     | 具体措施                                                    |
| ------- | ------------------------------------------------------- |
| 5 层速率限制 | 全局 200/分钟、邮箱 5/小时+10/天、IP 3/分钟+20/小时+100/天、IP+邮箱 3/10分钟 |
| 发送冷却    | 60 秒内不可重复发送                                             |
| 校验防爆破   | 连续 5 次错误 → 冻结 10 分钟                                     |
| 自适应人机验证 | 异常行为自动触发 Turnstile CAPTCHA                              |

### 2.3 Redis 分布式熔断器

120 秒窗口内连续 5 次网络类异常触发 90 秒熔断，自动切换到备用模型，恢复后自动清除熔断状态。基于 Redis 实现，支持多实例部署。配合线性退避重试（350ms→700ms）和 15 秒超时控制。

### 2.4 Token 配额三级动态降级

每日 1000 万 token 预算，按日消耗量自动触发三级降级（70%/85%/95%），逐级收紧 token 预算（×0.85/×0.7/×0.5）和游客权限（L2 起游客拒绝），保护系统不被打满。Redis 异常时自动进入保护模式。

### 2.5 精细化调用控制

按角色（游客 10 次/天 vs 用户 45 次/天）和模块（陪练 12/30、解析 30/7、简历 5/不可用、每日一句 16/5）分配不同调用次数，配合冷却时间（用户 3 秒、游客 5 秒、简历 10 秒），防止滥用。

### 2.6 Agent 两阶段架构

AI 回答 + 后台智能搜索关联题库/笔记补充。7 个工具注册体系，消息列表模式绕过模板引擎避免 JSON 解析错误，智能关键词提取，双保险搜索策略。

### 2.7 学习进度上下文注入

AI 教练知道用户做了哪些题、各分类掌握程度，提供个性化建议。查询 `user_question_mastery` 表 → 按分类统计掌握数量 → 提取最近 5 道题目标题 → 注入 Coach Prompt。

### 2.8 Redis 位图签到

每月每用户仅占 31 bit，连续签到回溯 3660 天，400 天 TTL 自动过期。每次登录自动触发，无需手动点击。

### 2.9 异步愿望 AI 审核流水线

8 个 Redis Key 协同 + 3 种定时任务守护（3 秒/30 秒/60 秒）+ RPOPLPUSH 防崩溃 + 分布式锁续期 + 3 次重试死信队列 + AI 情绪颜色映射。

### 2.10 多级缓存体系

题库和笔记模块采用 Redis 多级缓存（列表 2 分钟/详情 10 分钟/随机池 5 分钟/分类 30 分钟），SET NX 分布式锁防击穿（3 秒 + 3 次 40ms 自旋）+ TTL 随机抖动 `[0, TTL/5]` 防雪崩。

### 2.11 浏览量缓冲

浏览量永不直接写库，走 Redis 三层缓冲（Hash 累积 + String 总览 + ZSet 热度），每 15 秒定时批量刷库，每次最多 200 条，避免高并发下行锁争用。

### 2.12 IP 伪造防护

X-Forwarded-For 取最后一个非私有 IP 地址，防止攻击者通过伪造 IP 绕过限流和锁定。

### 2.13 全容器化部署

5 个容器通过 Docker Compose 编排，Nginx 内置反向代理 + CSP 安全头 + SSE 优化配置，前后端分离部署。

### 3.2 后端分层结构

```
┌─────────────────────────────────────────────────┐
│ Controller 层（HTTP 接入）                       │
│ - 只做三件事：参数绑定、调用 Application Service、 │
│   封装 Result 返回                               │
│ - 不包含任何业务逻辑                             │
│ - 使用 @Valid 做声明式参数校验                   │
├─────────────────────────────────────────────────┤
│ Application Service 层（业务编排）                │
│ - 权限判断、状态流转、参数归一化                 │
│ - 调用多个 support/infra 组件完成业务            │
│ - 命名规范：*ApplicationService                 │
│ - 事务注解 @Transactional 放在此层               │
├─────────────────────────────────────────────────┤
│ Support / Infra 层（基础设施）                    │
│ - Redis 操作封装                                 │
│ - MyBatis-Plus Mapper 查询                       │
│ - AI 模型调用（AiModelGateway）                  │
│ - 配额策略（AiQuotaPolicy）                      │
│ - 审计记录（AiCallAuditService）                 │
│ - 缓存支持（QuestionReadCacheSupport）           │
├─────────────────────────────────────────────────┤
│ Mapper 层（MyBatis-Plus）                        │
│ - 数据访问，继承 BaseMapper<T>                  │
│ - 复杂查询通过 LambdaQueryWrapper 构建          │
├─────────────────────────────────────────────────┤
│ Entity / DTO / VO 层                             │
│ - Entity：与数据库表一一映射                     │
│ - DTO：请求参数封装                             │
│ - VO：视图模型，只返回前端需要的数据            │
│ - Assembler：Entity → VO 转换                   │
└─────────────────────────────────────────────────┘
```

**关键设计规范**：

- 新业务优先在 `module/*/service/*ApplicationService` 中编写，避免逻辑堆积在 Controller 中
- Controller 层注入的是 ApplicationService，不是 Mapper 或 Support 类
- Entity 与 VO 分离：Entity 不直接返回前端，防止过度暴露字段
- 跨模块调用：ApplicationService 可以调用其他模块的 ApplicationService，不绕道 Controller

## 四、后端安全架构

### 4.1 核心设计理念

**用户端和管理端共用同一套后端服务**，不拆分为两个独立的后端应用。这样做的好处是减少重复代码，用户表、题库表、笔记表等核心实体天然共享，AI 能力、文件存储等基础设施无需两套。但这也带来了权限隔离的挑战。

解决方案是在 **Security 层通过单过滤器链 + URL 前缀路由 + Authority 权限表达式实现物理级的路径隔离**，而不是仅仅在业务代码中用 if-else 判断角色。

### 4.2 SecurityConfig 详解

**配置类**：`com.novaleap.api.config.SecurityConfig`（136 行）

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    // 管理端 API 的访问控制表达式
    // 需要同时满足两个条件：
    // 1. hasAuthority('PORTAL_ADMIN') —— 来自管理端 Portal
    // 2. hasRole('ADMIN') or hasRole('OPS') or hasRole('SUPER_ADMIN')
    private static final WebExpressionAuthorizationManager ADMIN_API_ACCESS =
            new WebExpressionAuthorizationManager(
                    "hasAuthority('PORTAL_ADMIN') and (hasRole('ADMIN') or hasRole('OPS') or hasRole('SUPER_ADMIN'))"
            );

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .cors(cors -> cors.configurationSource(corsConfigurationSource()))
            .csrf(csrf -> csrf.disable())  // 前后端分离 + JWT 无状态，CSRF 无意义
            .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .exceptionHandling(exception -> exception
                .authenticationEntryPoint((request, response, authException) ->
                    writeJsonIfNotCommitted(response, 401, Result.error(401, "请先登录")))
                .accessDeniedHandler((request, response, accessDeniedException) ->
                    writeJsonIfNotCommitted(response, 403, Result.error(403, "无访问权限"))))
            .authorizeHttpRequests(authz -> authz
                // ===== 公共接口（无需任何认证） =====
                .requestMatchers(HttpMethod.POST,
                    "/api/auth/login", "/api/auth/register", "/api/auth/guest",
                    "/api/auth/password/reset", "/api/auth/email/send-code",
                    "/api/admin/auth/login"
                ).permitAll()
                .requestMatchers(HttpMethod.GET,
                    "/api/leaderboard/**", "/api/questions", "/api/questions/categories",
                    "/api/questions/random", "/api/questions/{id}", "/api/questions/{id}/answer",
                    "/api/wishes", "/api/wishes/{id}/comments",
                    "/api/ai/question/{id}/explain", "/api/ai/quote/daily"
                ).permitAll()
                .requestMatchers(HttpMethod.POST,
                    "/api/analytics/visit", "/api/questions/{id}/view", "/api/wishes/{id}/like"
                ).permitAll()
                .requestMatchers("/ws/**").permitAll()
                .requestMatchers("/api/admin/auth/**").permitAll()
                // ===== 管理端 API：需要双因子权限 =====
                .requestMatchers("/api/admin/**").access(ADMIN_API_ACCESS)
                // ===== 其余所有接口要求认证 =====
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }

    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        // 从环境变量读取允许的域名列表（逗号分隔）
        // 生产环境必须配置，为空时启动失败
        if (allowedOrigins == null || allowedOrigins.isBlank()) {
            throw new IllegalStateException(
                    "NOVA_ALLOWED_ORIGINS is not configured. Set it to your frontend domain(s) in production.");
        }
        // ...
    }
}
```

**CORS 配置细节**：

- 支持从 `NOVA_ALLOWED_ORIGINS` 环境变量读取多个 origin（逗号分隔）
- 生产环境必须设置为前端的实际域名，不允许使用通配符
- 允许的方法：GET, POST, PUT, PATCH, DELETE, OPTIONS
- 暴露的响应头：Authorization, Content-Disposition
- `allowCredentials = false`（不使用 Cookie 认证）
- 预检请求缓存 1 小时

### 4.3 JWT 认证过滤链

**过滤器**：`com.novaleap.api.security.JwtAuthenticationFilter`

继承自 `OncePerRequestFilter`，保证每次请求只执行一次过滤：

```
请求到达
    │
    ▼
从 Header 提取 "Authorization: Bearer <token>"
    │
    ├── 无 Token → 跳过，交给 Spring Security 后续处理
    │               （公开接口通过 permitAll 放行，非公开接口返回 401）
    │
    ▼
解析 JWT Token（jjwt 库，HMAC-SHA256 签名）
    │
    ├── Token 过期/签名无效 → SecurityContextHolder.clearContext()
    │   本次更新：日志记录失败详情，便于排查认证问题
    │
    ▼
Token 状态校验（AuthTokenStateSupport）
    │
    ├── Token 已被手动失效（用户注销、重新登录、管理员踢下线）
    │   → SecurityContextHolder.clearContext()
    │
    ▼
提取 Claims：
    - username（sub）
    - role（USER / ADMIN / OPS / SUPER_ADMIN / GUEST）
    - portal（CLIENT / ADMIN）
    │
    ▼
构建 Spring Security Authentication：
    - principal = username
    - authorities = ROLE_{role} + PORTAL_{portal}
    │
    ▼
设置 SecurityContextHolder.getContext().setAuthentication()
    │
    ▼
放行到 Controller
```

**JWT 配置**：

- 签名算法：HMAC-SHA256（`Keys.hmacShaKeyFor`）
- 密钥：通过 `NOVA_JWT_SECRET` 环境变量配置（建议 256 位以上）
- 过期时间：**7 天**（604,800,000 ms，通过 `NOVA_JWT_EXPIRATION` 配置）
- Token 携带 claims：`sub`（用户名）、`role`（角色）、`portal`（入口 CLIENT/ADMIN）、`iat_ms`（毫秒级签发时间，用于精确状态校验）

### 4.4 AuthPortal 双 Portal 机制

`AuthPortal` 枚举是 NovaLeap 安全体系中非常核心的设计。它定义了 `CLIENT` 和 `ADMIN` 两个入口，每个入口有独立的：

| 属性               | CLIENT            | ADMIN            |
| ---------------- | ----------------- | ---------------- |
| claimValue()     | `"CLIENT"`        | `"ADMIN"`        |
| authority()      | `"PORTAL_CLIENT"` | `"PORTAL_ADMIN"` |
| redisNamespace() | `"client"`        | `"admin"`        |

JWT Token 中携带 `portal` 字段，过滤器解析后转化为 `PORTAL_CLIENT` 或 `PORTAL_ADMIN` 权限。SecurityConfig 中的表达式确保了管理端 API 必须要有 `PORTAL_ADMIN` 权限才能访问，而用户端 Token 只有 `PORTAL_CLIENT`，从根本上杜绝了权限越界。

即使攻击者获取了用户端的 JWT Token，也无法用它访问任何 `/api/admin/` 接口——因为 `PORTAL_CLIENT ≠ PORTAL_ADMIN`。

### 4.5 AuthTokenStateSupport（Token 状态管理）

**实现类**：`com.novaleap.api.module.auth.support.AuthTokenStateSupport`

当用户执行以下操作时，已签发的 JWT Token 会被标记为失效：

- 主动注销（`/api/auth/logout`）
- 重置密码
- 管理员后台手动下线用户

**实现原理**：在 Redis 中记录 `nova:auth:token:invalid-after:{username}`，值为时间戳。JWT 过滤器中对比 Token 的 `iat`（签发时间）是否晚于该时间戳，如果早于则判定失效。

这个机制解决了 JWT 天然无法被服务端主动撤销的问题——不需要维护一个庞大的黑名单，只需要一个键值对就能实现。

**本次更新（fail-safe）**：异常解析时返回 `false`（原为 `true`）。Token 状态无法判定时视为无效而非有效，防止 Redis 异常导致旧 Token 无限期可用。

### 4.6 密码加密与校验

**工具类**：`AuthPasswordSupport.java`

- 使用 `BCryptPasswordEncoder`（Spring Security 内置）加密存储，密码哈希以 `$2a$` / `$2b$` / `$2y$` 开头
- 密码长度限制 **6~20** 位
- 注册时密码与确认密码必须一致，使用 `@Valid` Jakarta Validation 注解进行参数校验
- **遗留密码自动迁移**：如果数据库中的密码不是 BCrypt 格式（`$2a$` 开头），会用明文匹配后自动编码为 BCrypt 并更新数据库，实现无感升级
- 昵称长度限制 **2~50** 位（含中文）

```

### 5.1 登录方式

系统支持三种登录方式：

#### 5.1.1 邮箱 + 密码登录

最基础的登录方式。用户输入邮箱和密码，后端通过 `BCryptPasswordEncoder` 校验密码哈希。整个流程走 `SharedAuthLoginService.authenticateByPassword()`：

```
用户提交邮箱 + 密码
    │
    ▼
AuthRateLimitSupport.checkLoginRateLimit()  →  限流检查
    │  用户端：每分钟 30 次
    │  管理端：每分钟 12 次
    │  Key: nova:auth:{portal}:login:req:{ip}:{分钟桶}
    ▼
AuthRateLimitSupport.isLoginLocked()         →  是否被锁定
    │  连续 5 次失败 → 锁定
    │  用户端锁 10 分钟，管理端锁 15 分钟
    │  Key: nova:auth:{portal}:login:fail:{ip}:{username}
    ▼
TurnstileVerifier.verifyIfEnabled()          →  人机验证（可选）
    │  通过 Cloudflare Turnstile 后端验证
    │  通过 nova.turnstile.enabled 控制开关
    ▼
AuthService.authenticate()                   →  查询用户 + BCrypt 校验密码
    │
    ├── 成功 → AuthRateLimitSupport.clearLoginFailures() 清除失败记录
    │          JwtUtils.generateToken() 生成 JWT（含 role + portal）
    │          AuthCheckinSupport.resolveCheckinPayload() 自动签到
    │          SharedAuthLoginService.buildAuthResult() 返回 Token + 用户信息
    │
    └── 失败 → AuthRateLimitSupport.recordLoginFailure() 记录失败次数
               throws IllegalArgumentException("账号或密码错误")
```

#### 5.1.2 邮箱验证码登录

用户先请求发送验证码到邮箱，再用验证码登录。适用于忘记密码或不想记密码的场景：

```
用户请求发送验证码
    │
    ▼
EmailService.sendVerificationCode()
    ├── 校验邮箱格式（正则：^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$）
    ├── enforceSendRateLimits()  →  七层限流检查
    ├── 60 秒冷却检查（同一邮箱不可重复发送）
    ├── 判断邮箱是否已注册（登录/重置需要已注册，注册需要未注册）
    ├── 自适应人机验证：IP 分钟请求≥2 或组合≥2 或邮箱小时≥3 时触发 Turnstile
    ├── 生成 6 位随机数字验证码（SecureRandom）
    ├── 存入 Redis（Key: nova:auth:verify:code:{type}:{email}，TTL 5 分钟）
    └── 通过 Resend API 发送邮件

用户提交邮箱 + 验证码
    │
    ▼
EmailService.verifyCodeWithPolicy()  →  校验验证码有效性
    │
    ├── 检查是否被冻结（5 次错误→冻结 10 分钟）
    ├── 检查验证码是否存在
    ├── 比对验证码
    │   ├── 通过 → consumeCode() 消耗验证码（防止重复使用）
    │   │          SharedAuthLoginService.authenticateByVerifiedIdentity()
    │   │          → 查询用户 → 生成 JWT → 返回结果
    │   │
    │   └── 失败 → 递增失败计数
    │              5 次失败 → 删除验证码 + 冻结 10 分钟
    │              throws IllegalArgumentException("验证码错误或已过期")
```

#### 5.1.3 游客登录

用户无需任何注册操作，调用 `/api/auth/guest` 即可获取一个临时身份：

```
ClientAuthApplicationService.guestLogin()
    │
    ▼
AuthService.guestLogin()
    ├── 生成 guest_{uuid} 格式的用户名
    ├── 在 User 表中插入游客记录（role = 'GUEST'）
    ├── JwtUtils.generateToken() 生成 JWT（role = 'GUEST', portal = 'CLIENT'）

游客有独立的角色 `GUEST`，在 AI 配额系统中受到严格限制（每天最多 10 次调用，L2 降级后全部拒绝），且无法使用需要数据库用户的功能（如写笔记、简历优化）。

### 5.2 注册流程

```
用户提交邮箱 + 密码 + 确认密码 + 邮箱验证码 + 昵称

    ▼
参数校验（@Valid）
    ├── 邮箱格式合法
    ├── 密码长度 6~20 位

    └── 同意用户协议
    │

### 5.3 密码重置

```
用户提交邮箱 + 新密码 + 邮箱验证码
    │
    ▼
校验邮箱验证码 → 通过
    │
    ▼

    ├── BCryptPasswordEncoder.encode(newPassword) 加密新密码

### 5.4 邮箱验证码系统（七层防刷）

**实现类**：`EmailService.java`（510 行）
**邮件提供商**：Resend（`resend-java` SDK，通过 `nova.mail.resend.api-key` 配置）
**验证码生成**：`SecureRandom` 生成 6 位数字
**限流实现**：通过 `DefaultRedisScript` 执行内嵌 Lua 脚本，保证 INCR + EXPIRE 原子性（[EmailService.java:66-78](nova-backend/src/main/java/com/novaleap/api/module/auth/service/EmailService.java#L66-L78)）    

```lua
-- 窗口计数器 Lua 脚本（内嵌在 Java 中）
local current = redis.call('INCR', KEYS[1])
if current == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end
local ttl = redis.call('TTL', KEYS[1])
if ttl < 0 then ttl = tonumber(ARGV[1]) end
if current > tonumber(ARGV[2]) then return {0, current, ttl} end
return {1, current, ttl}
```

脚本返回 `{allowFlag, currentCount, ttlSeconds}`，allowFlag=1 表示允许、0 表示拒绝。拒绝时返回剩余 TTL，前端可提示"请 X 秒后再试"。

#### Redis Key 完整体系（11 个 Key）

| 分类    | Key                                                  | 用途       | TTL   |
| ----- | ---------------------------------------------------- | -------- | ----- |
| 验证码   | `nova:auth:verify:code:{type}:{email}`               | 存储验证码本体  | 5 分钟  |
| 冷却    | `nova:auth:verify:cooldown:{type}:{email}`           | 发送冷却标记   | 60 秒  |
| 失败计数  | `nova:auth:verify:fail:{type}:{email}`               | 校验失败计数   | 5 分钟  |
| 冻结    | `nova:auth:verify:freeze:{type}:{email}`             | 校验冻结标记   | 10 分钟 |
| 邮箱限流  | `nova:auth:verify:send:email:{type}:1h:{email}`      | 单邮箱小时级   | 1 小时  |
| 邮箱限流  | `nova:auth:verify:send:email:{type}:24h:{email}`     | 单邮箱天级    | 24 小时 |
| IP 限流 | `nova:auth:verify:send:ip:1m:{ip}`                   | 单 IP 分钟级 | 1 分钟  |
| IP 限流 | `nova:auth:verify:send:ip:1h:{ip}`                   | 单 IP 小时级 | 1 小时  |
| IP 限流 | `nova:auth:verify:send:ip:24h:{ip}`                  | 单 IP 天级  | 24 小时 |
| 组合限流  | `nova:auth:verify:send:pair:{type}:10m:{ip}:{email}` | IP+邮箱组合  | 10 分钟 |
| 全局限流  | `nova:auth:verify:send:global:1m`                    | 全局发送限流   | 1 分钟  |

#### 速率限制阈值（使用 Lua 脚本原子操作）

| 维度      | 限制    | 窗口    | 超过后提示            |
| ------- | ----- | ----- | ---------------- |
| 全局      | 200 次 | 1 分钟  | "请求过于频繁，请稍后重试"   |
| 单邮箱     | 5 次   | 1 小时  | "该邮箱发送次数过多"      |
| 单邮箱     | 10 次  | 24 小时 | "该邮箱今日发送次数已达上限"  |
| 单 IP    | 3 次   | 1 分钟  | "请求过于频繁"         |
| 单 IP    | 20 次  | 1 小时  | "当前网络请求过于频繁"     |
| 单 IP    | 100 次 | 24 小时 | "当前网络今日请求次数已达上限" |
| IP+邮箱组合 | 3 次   | 10 分钟 | "当前操作过于频繁"       |

#### 安全策略

- 验证码有效期 **5 分钟**
- 验证码发送后 **60 秒冷却**，同一邮箱不可重复发送
- 验证码使用后 `consumeCode()` 立即消耗，不能重复使用
- 校验连续错误 **5 次** → 自动**冻结 10 分钟**，期间任何校验直接返回锁定状态
- **自适应人机验证**：当 IP 分钟请求 ≥ 2、或 IP+邮箱组合 ≥ 2、或邮箱小时请求 ≥ 3 时，下次发送强制要求 Turnstile CAPTCHA 验证
- 验证码类型：`register`（注册）、`login`（登录）、`reset`（重置密码），不同类型对邮箱状态有不同要求

#### Lua 窗口计数器脚本

```lua
local current = redis.call('INCR', KEYS[1])
if current == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end
local ttl = redis.call('TTL', KEYS[1])
if ttl < 0 then ttl = tonumber(ARGV[1]) end
if current > tonumber(ARGV[2]) then return {0, current, ttl} end
return {1, current, ttl}
```

- 使用 Lua 脚本保证 INCR + EXPIRE 的原子性
- 返回 `{allowFlag, currentCount, ttlSeconds}`，其中 allowFlag=1 表示允许、0 表示拒绝
- 拒绝时返回剩余 TTL（秒数），前端可提示"请 X 秒后再试"

### 5.5 签到功能

**实现类**：`AuthCheckinSupport.java`（80 行）

签到使用 **Redis 位图（Bitmap）** 实现：

```
Key 结构：nova:checkin:{userId}:{yyyyMM}
例如：nova:checkin:42:202607

签到操作：将当天对应的 bit 设为 1
redisTemplate.opsForValue().setBit(key, dayOfMonth - 1, true);

查询连续签到天数：从今天开始往前数，遇到 0 就停
```java
for (int i = 0; i < 3660; i++) {  // 最多回溯 10 年
    if (!isSigned(userId, cursor)) break;
    streak++; cursor = cursor.minusDays(1);
}
```

```
Redis 位图实现签到的好处：
- **存储极省**：每月每个用户只占 31 bit（不到 4 字节），即使用户量上万也占用极小
- **查询高效**：`GETBIT` 是 O(1) 操作，连续签到查询最多 3660 次 GETBIT
- **自动过期**：位图 Key 设置 **400 天 TTL**，保证足够回溯周期但不会永久堆积
- **天然支持跨月统计**：Key 按 `{userId}:{yyyyMM}` 分片，可以分别查询不同月份数据

签到在用户登录时**自动触发**（`buildAuthResult()` 携带 `markLoginCheckin=true` 参数），无需用户手动点击。

### 5.6 登录安全策略

**登录限流**（`AuthRateLimitSupport`）

基于 Redis 的滑动窗口实现，每分钟一个 bucket：

| 限制项 | 用户端 | 管理端 |
|--------|-------|-------|
| 登录请求限流 | 每分钟 30 次 | 每分钟 12 次 |
| 连续失败锁定阈值 | 5 次 | 5 次 |
| 锁定时长 | 10 分钟 | 15 分钟 |

限流的 Key 设计：`nova:auth:{portal}:login:req:{ipKey}:{分钟桶}`

锁定分两个维度：
- **IP + 用户名双维度锁定**：Key 为 `nova:auth:{portal}:login:fail:{ipKey}:{usernameKey}`
- 防止同一 IP 暴力破解不同账号，也防止同一账号被分布式暴力破解

**注册限流**：每 IP 每小时最多 20 次注册

### 6.3 熔断器（Circuit Breaker）

**实现类**：`AiModelGateway.java`（848 行）
**存储**：Redis（支持多实例部署，不是本地内存）

熔断器的设计参考了分布式系统中经典的 Circuit Breaker 模式，但做了简化——只区分"关闭"和"开启"两个状态（没有半开状态），因为半开状态在 AI API 场景下的实现复杂度较高，而且 90 秒后自动恢复已经足够。

**Redis Key 设计**：

- 失败计数器：`nova:ai:circuit:fail:{module}:{model}`（TTL 120 秒）
- 熔断标记：`nova:ai:circuit:open:{module}:{model}`（TTL 90 秒）

**熔断流程**：

```
正常调用
    │
    ├── 成功 → 清除失败计数和熔断标记
    │
    └── 失败
        │
        ├── 可重试异常？（网络超时/连接重置/502/503/504）
        │   ├── 是 → Redis 递增失败计数器，TTL 120 秒
        │   │        │
        │   │        └── 连续失败 ≥ 5 次？
        │   │            ├── 是 → 写入熔断标记，TTL 90 秒
        │   │            │        熔断期间所有请求切到备用模型
        │   │            │        90 秒后自动恢复
        │   │            │
        │   │            └── 否 → 继续用主模型
        │   │
        │   └── 不可重试异常？（429/402/配额不足/rate limit）
        │       → 直接切到备用模型，不触发熔断
        │
        ▼
    重试机制 (executeWithRetry)
        ├── 最大重试 2 次（共 3 次尝试）
        ├── 线性退避：350ms → 700ms（retryBackoffMs × attempt）
        └── 仅可重试异常才重试
```

**配置参数**：

| 参数                          | 默认值   | 说明                      |
| --------------------------- | ----- | ----------------------- |
| `circuit-window-seconds`    | 120   | 失败计数窗口（秒）               |
| `circuit-failure-threshold` | 5     | 熔断触发阈值（连续失败次数）          |
| `circuit-open-seconds`      | 90    | 熔断持续时间（秒）               |
| `request-timeout-ms`        | 15000 | 单次请求超时（毫秒）              |
| `max-retry-attempts`        | 2     | 最大重试次数（初始 1 次 + 重试 2 次） |
| `retry-backoff-ms`          | 350   | 重试退避基础间隔（毫秒），线性增长       |

### 6.4 SSE 流式输出

**后端实现**（`AiModelGateway.java`）：

- 创建 `SseEmitter`，超时 **120 秒**
- 使用 Spring AI Reactive 流 API（`stream().chatResponse().subscribe()`）异步订阅
- **逐字符发送**：每个 SSE event 携带一个 Unicode 码点，格式 `{"type":"delta","content":"某"}`
- `onTimeout` 回调触发 `finishModelStream()`，确保流正常关闭
- 使用 `AtomicBoolean` 确保 `finishModelStream()` 只执行一次
- 无内容时发送预设的 fallback 文本
- 流结束时发送 `{"type":"done"}` 事件

**前端实现**（`useSSE.js`，309 行）：

- 基于原生 `fetch` + `ReadableStream` 手动解析 SSE 协议
- 支持 `data:` 前缀行和空行分隔的标准 SSE 格式
- 兼容 OpenAI 格式（`{choices:[{delta:{content}}]}`）和自定义格式
- 收到 `delta` 后积累到 `pendingChunks` 缓冲区
- 通过 **18ms 定时器** 分批渲染，每次取 `max(1, min(26, ceil(pending/24)))` 个字符，实现平滑打字机效果
- 两个超时：首包 **6 秒** + 整体 **25 秒**
- 网络错误时自动重试（默认 **1 次**，**700ms** 基础退避，仅重试 HTTP 408/429/502/503/504）
- 错误消息中文友好化（"failed to fetch" → "网络连接失败，AI 服务暂时不可用"）
- 请求级 `activeRequestId` 计数器，忽略过期响应

**降级链路**：SSE 流失败 → 同步 `callModel()` 兜底 → `streamStaticAnswer()` 模拟流式输出（48 字符/块，22ms 间隔）

### 6.6 Token 配额与动态降级系统

#### 配额体系三层结构

```
AiQuotaProperties（配置层）→ AiQuotaPolicy（策略层）→ AiQuotaUsageSupport（存储层）
```

#### 每日 Token 预算

| 配置项                     | 默认值                    | 说明                 |
| ----------------------- | ---------------------- | ------------------ |
| `daily-token-limit`     | **10,000,000**（1000 万） | 全局每日总 token 预算     |
| `per-model-token-limit` | **5,000,000**（500 万）   | 单个模型每日 token 预算    |
| `model-usage-ttl-days`  | 2                      | 模型用量统计 TTL         |
| `min-response-tokens`   | 128                    | 单次调用最小预算（低于此值直接拒绝） |

#### 动态降级级别

根据日消耗量百分比自动计算降级级别：

```
日消耗量 = (当日已用 tokens / 每日总预算) × 100%

日消耗 < 70%  ──▶ Level 0（正常）   ：预算 ×1.0，游客可用 10 次
70% ≤ 日消耗 < 85%  ──▶ Level 1（预警） ：预算 ×0.85，游客从 10 次降为 6 次
85% ≤ 日消耗 < 95%  ──▶ Level 2（保护） ：预算 ×0.70，游客全部拒绝
日消耗 ≥ 95%  ──▶ Level 3（极限） ：预算 ×0.50，仅用户可用 SOLVER/COACH
```

**阈值校验**：`validateThresholdOrder()` 确保 `warningRatio (0.70) < protectRatio (0.85) < limitRatio (0.95)`，配置错误时项目启动失败。

**Redis 异常兜底**：`getCurrentDegradeLevel()` 捕获 Redis 异常时返回 L2（保护模式），防止 Redis 宕机后配额检查失效。

#### 每次调用的 Token 预算

| 模块           | 基础预算 | 受降级影响 | 说明               |
| ------------ | ---- | ----- | ---------------- |
| RESUME（简历分析） | 5000 | ❌     | 简历文本量大，降级后预算不够分析 |
| SOLVER（题目解析） | 1200 | ✅     | 单道题目的 AI 解析      |
| COACH（AI 陪练） | 1000 | ✅     | 多轮对话的每次回复        |
| CHAT（每日一句）   | 800  | ✅     | 单句励志语录生成         |

降级后预算公式：`最终预算 = min(基础预算 × 降级系数, 模型剩余配额, 全局剩余配额)`

#### 模块调用次数限制

| 模块           | 游客      | 已注册用户  |
| ------------ | ------- | ------ |
| COACH（AI 陪练） | 3 次/天   | 12 次/天 |
| SOLVER（题目解析） | 7 次/天   | 30 次/天 |
| RESUME（简历分析） | **不可用** | 5 次/天  |
| CHAT（每日一句）   | 5 次/天   | 16 次/天 |

**总次数限制**：

- 已注册用户：每天最多 **45 次** AI 调用（所有模块合计）
- 游客：正常时 **10 次/天**，Level ≥ 1 时 **6 次/天**

#### 调用冷却时间

| 角色    | 默认冷却    | 简历分析（RESUME） |
| ----- | ------- | ------------ |
| 游客    | **5 秒** | 10 秒         |
| 已注册用户 | **3 秒** | 10 秒         |

冷却 Key：`ai:usage:cooldown:{identifier}:{module}`

#### 配额校验完整流程

`AiLimitServiceImpl.checkLimit()` 的执行顺序严格定义：

```
请求到达
    │
    1. 身份识别
    ├── 未识别（anonymous）→ ❌ 拒绝
    │
    2. 降级级别 ≥ 3？（极限模式）
    ├── 是 → 仅允许已登录用户的 SOLVER/COACH，游客请求全部拒绝
    │
    3. 降级级别 ≥ 2？（保护模式）
    ├── 是 → 游客全部拒绝
    │
    4. 冷却检查（inCooldown）
    ├── True → ❌ 拒绝（冷却中）
    │
    5. 总次数检查（totalLimit）
    ├── 超额 → ❌ 拒绝
    │
    6. 模块次数检查（moduleLimit）
    ├── 超额 → ❌ 拒绝
    │
    7. ✅ 全部通过 → 增加计数 + 设置冷却标记 → 放行
```

## 九、笔记模块

### 9.1 笔记状态机

```
待审核 (0) ──审核通过──▶ 已发布 (1)
    │                      │
    └──审核拒绝──▶ 拒绝 (2)  └──用户编辑──▶ 待审核 (0)
```

- 新建笔记默认状态为 `PENDING (0)`，需要管理端审核
- 审核通过后状态变为 `APPROVED (1)`，公开可见
- 审核拒绝后可查看拒绝原因，修改后重新提交
- 笔记编辑后状态重置为 `PENDING`，重新进入审核流程

### 9.2 AI 双检内容审核

笔记提交时，会经过两道内容安全防线：

1. **硬编码敏感词过滤**：`HARD_BANNED_WORDS` 集合（19 个），包含毒品、赌博、诈骗、色情、暴力等类别，使用 `contains()` 匹配（不区分大小写）
2. **AI 语义审核**：调用 `AiNoteWorkflowSupport.moderateNote()`，AI 模型输出 `{"approved":true/false,"reason":"..."}`
3. 审核来源标记 `audit_source`：`RULES`（规则拦截）/ `AI`（AI 审核）/ `MANUAL`（管理员手动）

### 9.3 笔记缓存

| 缓存项     | Key 模式                                                | TTL   |

| 笔记详情    | `note:detail:{id}`                                    | 10 分钟 |

## 十一、星愿墙

### 11.1 异步 AI 审核流水线

```
用户发布愿望 → INSERT status=0 (PENDING)
    │  （事务提交后通过 TransactionSynchronizationManager 入队）
    ▼
Redis List: nova:wish:pending
    │
    ▼（每 3 秒消费，每次最多 5 条）
RPOPLPUSH → 处理队列（防止消费者崩溃丢失数据）
    │ 获取分布式锁（TTL 180 秒，每 45 秒续期）
    ▼
AI 分析情绪 → happy/hopeful/confused/anxious
    │
    ├── 成功 → 设置 color/floatSpeed/posX/posY → status=1 (APPROVED)
    └── 失败 → 重试（最多 3 次）
                └── 3 次均失败 → 死信队列 nova:wish:dead
```

### 11.2 Redis Key 体系（8 个 Key 协同工作）

| Key                                 | 类型        | 用途                |
| ----------------------------------- | --------- | ----------------- |
| `nova:wish:pending`                 | List      | FIFO 待审核队列        |
| `nova:wish:pending:index`           | Set       | 去重索引，防止重复入队       |
| `nova:wish:processing:list`         | List      | RPOPLPUSH 安全缓冲区   |
| `nova:wish:processing:zset`         | SortedSet | 处理开始时间戳（用于超时恢复）   |
| `nova:wish:processing:lock:{id}`    | String    | 分布式处理锁（TTL 180 秒） |
| `nova:wish:processing:recover:lock` | String    | 恢复操作锁             |
| `nova:wish:retry:{id}`              | String    | 重试计数器（TTL 1 天）    |
| `nova:wish:dead`                    | Set       | 死信标识集合            |

**实现类**：`WishConsumerTask.java`（385 行）

## 十四、浏览量缓冲

**实现类**：`ContentViewFlushTask.java`（118 行）

浏览量永不直接写库，走三层 Redis 缓冲：

```
用户浏览
    │
    ▼
Redis Hash: nova:question:view:pending (delta +1)    ← 累积增量
Redis String: nova:question:view:total:{id} (total +1) ← 已解析总览
Redis ZSet: nova:question:hot:zset (score +1)         ← 热度排行
    │
    │  每 15 秒定时任务（@Scheduled(fixedDelay = 15000)）
    │  每次最多 200 条
    │
    ▼
UPDATE questions SET view_count = IFNULL(view_count, 0) + <delta>
WHERE id = <id> AND status = 1

UPDATE notes SET view_count = IFNULL(view_count, 0) + <delta>
WHERE id = <id> AND deleted = 0
```

**为什么不是实时写库**：Redis Hash 操作是 O(1) 的内存操作，高并发下性能极好。每 15 秒批量刷库将大量浏览事件合并为几条 UPDATE 语句，避免 MySQL 行锁争用。

## 十七、前端架构

### 17.1 路由与代码分割

**15 个路由全量懒加载**（`import()`），按需生成独立 chunk：

| 路由                                                | 组件                          | 需要认证 |
| ------------------------------------------------- | --------------------------- | ---- |
| `/login` `/register` `/forgot-password`           | Login.vue（同一组件，path 切换 tab） | 否    |
| `/terms` `/privacy`                               | Terms.vue / Privacy.vue     | 否    |
| `/`                                               | Layout.vue → Home.vue       | 是    |
| `/questions` `/notes` `/resume` `/coach`          | 各视图                         | 是    |
| `/wishes` `/game` `/me` `/leaderboard` `/profile` | 各视图                         | 是    |

**手动代码分割**（`vite.config.js` `manualChunks`）：

- `vue-vendor`：`vue` + `vue-router` + `pinia`
- `markdown`：`marked`

**路由预加载**（`warmRouteChunks`）：导航完成后，通过 `requestIdleCallback`（1800ms 超时回退）预加载所有认证路由的 chunk。慢网络环境跳过（检测 `navigator.connection.effectiveType` 为 slow-2g/2g/3g 或 `saveData=true`）。

**路由守卫**：

- `beforeEach`：从 `sessionStorage` 读取 Token（`localStorage` 迁移兼容），检查 `requiresAuth` + `GUEST` 角色
- 游客角色不重定向登录页（允许在登录页注册/切换账号）
- 120ms 防闪烁加载指示器
- `afterEach`：触发预加载 + 访客上报 + 停止加载

### 17.2 Pinia Store

**`auth` Store**（Setup Store，87 行）：

- 状态：`token`、`user`（从 `sessionStorage` 读取，`localStorage` 迁移兼容）
- 计算属性：`isLoggedIn`、`isAdmin`、`isGuest`、`nickname`、`avatar`
- 令牌键：`nova_token`，用户键：`nova_user`
- Token 优先存 `sessionStorage`（标签页关闭即清除），旧 `localStorage` 数据自动迁移

**`ui` Store**（Setup Store，67 行）：

- 状态：`sidebarCollapsed`、`isDarkMode`（`localStorage` 持久化键 `nova_theme`）、`globalLoading`、`notifications`
- 暗色模式：toggle 时设置 `document.documentElement.classList` 'dark'

**`resource-cache` Store**（Options Store，121 行）：

- 命名空间化的内存缓存（list + detail 二级）
- 请求去重：模块级 `Map` 跟踪进行中的 Promise，相同 key 的并发请求共享同一 Promise

### 17.3 核心 Composables

**`useSSE()`**（309 行）：SSE 流式消费。详见第 6.4 节。在 `Coach.vue` / `Resume.vue` / `QuestionBank.vue` 中使用。

**`useRequest()`**（147 行）：

- 基于 `fetch` + `AbortController`，默认 12 秒超时
- 自动附加 `Authorization: Bearer` 头
- HTTP 401 清除 token 并跳转 `/login`
- HTTP 408/429/502/503/504 可重试（指数退避）
- 提供 `api.get/post/put/delete/upload` 便捷方法

**`useAuth()`**：封装登录（密码/验证码两种模式）、游客登录、注册、登出、重置密码。支持对象传参和传统位置参数两种调用方式。登录后 `applyAuth()` 处理签到数据归一化。

**`useConfetti()`**（75 行）：基于 `canvas-confetti`，单例 canvas（z-index 9999），主线程运行。

**`useParallax()`**（40 行）：鼠标驱动的 3D 视差效果，每个元素独立系数 × 80 倍率。

**`useAutoPageRefresh()`**：页面可见性/焦点/`KeepAlive` 激活/定时器触发的自动刷新，默认节流 5000ms。

### 17.4 前端动画

- **HeartBurstBackground.vue**（334 行）：文本→粒子形变动画，在"张志琪"和"NovaLeap"间循环，最多 1100 粒子，`requestAnimationFrame` 驱动，鼠标互动影响粒子
- **Login.vue 粒子系统**：160 粒子 + 颜色轨迹 + 鼠标追踪眼睛动画（当前禁用）
- **Game.vue**：Canvas 2D 游戏"Light Leaper"，`requestAnimationFrame` 游戏循环

### 17.5 管理端（nova-admin）

| 维度  | 详情                                                   |
| --- | ---------------------------------------------------- |
| 路由  | 无懒加载，8 个路由                                           |

## 十九、安全增强汇总

### 19.1 IP 伪造防护（X-Forwarded-For）

`ClientRequestService.resolveClientIp()` 从"取第一个 IP"改为**取最后一个非私有 IP**（`lastNonPrivateIp()`）。原实现取 `X-Forwarded-For` 第一个 IP，攻击者可伪造 `X-Forwarded-For: 1.2.3.4, <real-ip>` 绕过 IP 限流。新实现取链中最后一个可路由地址，避免 IP 限流和锁定被绕过。

抽取 `isPrivateIp()` 方法统一判断私有 IP 范围（127/10/192.168/172.16-31/::1）。

### 19.2 CORS 配置启动校验

`allowedOrigins` 为空或空白时，`CorsConfigurationSource` 抛出 `IllegalStateException` 导致启动失败，防止因配置遗漏导致跨域安全风险。

### 19.3 JWT 校验日志

`JwtAuthenticationFilter` 增加日志记录：`log.warn("JWT validation failed: {} - {}", e.getClass().getSimpleName(), e.getMessage())`，便于排查认证失败原因。

### 19.4 Token 状态异常容错

`AuthTokenStateSupport.isTokenActive()` 异常时返回 `false`（原为 `true`），防止 Redis 异常导致失效 Token 继续可用。

### 19.5 配额 Redis 异常兜底

`AiLimitServiceImpl.getCurrentDegradeLevel()` 捕获 Redis 异常，返回 L2（保护模式），防止 Redis 宕机后配额检查失效导致 AI 被滥用。
