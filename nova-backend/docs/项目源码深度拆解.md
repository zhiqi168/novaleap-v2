# NovaLeap 后端项目源码深度拆解

## 扫描范围与结论

本次分析基于 `nova-backend/src/main/java/com/novaleap/api`、`src/main/resources`、`pom.xml`、`Dockerfile`、根目录 `docker-compose.yml` 的真实源码完成，不是按常见模板脑补。

先给你一个全局结论：

- 这是一个 **Spring Boot 3 + Spring Security + JWT + MyBatis-Plus + MySQL + Redis + Spring AI** 的 Java 后端项目。
- 项目一共扫描到 **15 个 Controller，73 个接口**。
- 架构风格不是完全统一的传统三层，也不是纯 DDD，而是 **“Controller + ApplicationService + BaseMapper + Support/Assembler + 少量传统 service/impl” 的混合结构**。
- 新业务更偏向 `module/*/service/*ApplicationService` 风格；老业务和跨模块能力仍然保留在 `service/impl`。
- Redis 在这个项目里不是简单做缓存，而是承担了 **验证码、限流、排行榜、AI 配额、签到位图、游客分析、愿望队列、头像、会话历史** 等多种状态型职责。
- 安全链路已经成型，但还存在几个典型“能跑但企业里会继续收紧”的点，比如：`logout` 形同空实现、若干接口的可见性控制不够严谨、配置里有敏感默认值。

# 第一部分：项目整体扫描（建立全局认知）

## 1. 项目目录结构

### 1.1 顶层后端目录

- `src/main/java/com/novaleap/api`
  作用：后端主业务源码入口。
  为什么放这里：Spring Boot 默认按主启动类所在包向下扫描 Bean，这样 `controller/config/service/module/security` 都能被自动扫描。
  请求是否参与：会，所有接口、配置、过滤、安全链都从这里进入。

- `src/main/resources`
  作用：放 `application.yml`、多环境配置、SQL 初始化脚本。
  为什么放这里：Spring Boot 启动时自动加载资源配置。
  请求是否参与：间接参与，决定数据库、Redis、JWT、AI、邮件等运行时行为。

- `docs`
  作用：已有接口文档和重构交付文档。
  为什么放这里：方便交付和二次维护。
  请求是否参与：不参与运行，但参与理解。

- `sql`
  作用：增量建表或修复 SQL。
  为什么放这里：用于开发部署时同步数据库结构。
  请求是否参与：不直接参与请求，但决定表结构是否支撑代码。

- `Dockerfile`
  作用：构建后端镜像。
  为什么放这里：便于容器化发布。
  请求是否参与：不参与业务逻辑，但参与部署链路。

- `pom.xml`
  作用：声明 Maven 依赖和构建插件。
  为什么放这里：Java 项目依赖入口。
  请求是否参与：不在请求期参与，但决定运行期能力。

### 1.2 Java 包级目录

- `common`
  真实文件：`Result.java`、`GlobalExceptionHandler.java`、`common/exception/*`
  作用：统一返回、统一异常、业务异常体系。
  为什么放这一层：这是横切能力，所有模块都可能依赖。
  上下层关系：Controller 返回 `Result`，Service 抛异常，`GlobalExceptionHandler` 兜底处理。
  请求是否参与：会。一次请求结束时几乎总会经过 `Result` 封装；异常时一定经过全局异常处理。

- `config`
  真实文件：`SecurityConfig.java`、`MybatisPlusConfig.java`、`UserUsernameUniqGuard.java`
  作用：安全链、MyBatis-Plus 分页插件、启动后用户名唯一性修复。
  为什么放这一层：配置类属于系统装配层，不属于具体业务模块。
  上下层关系：比 Controller/Service 更靠外层，先于业务生效。
  请求是否参与：`SecurityConfig` 和 `MybatisPlusConfig` 会参与；`UserUsernameUniqGuard` 只在启动时参与。

- `controller`
  真实文件：15 个 Controller。
  作用：接口入口层，只负责接收 HTTP 请求、绑定参数、调用服务、返回 `Result`。
  为什么放这一层：把 Web 层和业务层分开。
  上下层关系：上接前端请求，下调 `ApplicationService` 或传统 `Service`。
  请求是否参与：每个接口必经 Controller。

- `dto`
  真实文件：`LoginRequest`、`RegisterRequest`、`VisitTrackRequest`、`WishRequest` 等。
  作用：接前端请求体、少量返回对象。
  为什么放这一层：避免 Controller 直接接 Entity。
  上下层关系：Controller 用 DTO 接参，Service 读取 DTO 字段，VO 再负责返回。
  请求是否参与：会，Body 请求几乎都先封装成 DTO。

- `entity`
  真实文件：`User`、`Question`、`Note`、`Wish`、`CustomQuestionBank` 等。
  作用：数据库表映射对象。
  为什么放这一层：让 Mapper 直接和表结构对应。
  上下层关系：Mapper 查出来的是 Entity，Assembler 再转 VO。
  请求是否参与：会，Service 层对数据库读写就是基于 Entity。

- `mapper`
  真实文件：全部继承 `BaseMapper`，如 `UserMapper`、`QuestionMapper`、`WishMapper`。
  作用：数据访问层。
  为什么放这一层：把数据库操作从业务逻辑里拆开。
  上下层关系：Service 调 Mapper；Mapper 不关心 HTTP，不关心前端。
  请求是否参与：会，所有持久化请求都会走 Mapper。

- `module`
  作用：新风格业务模块聚合目录。
  为什么放这一层：比传统 `service/serviceImpl` 更容易按业务拆包。
  上下层关系：Controller 直接依赖 `module/*/service/*ApplicationService`，模块内部再细分 dto/vo/assembler/support。
  请求是否参与：会，是现在主要业务逻辑所在地。

- `security`
  真实文件：`JwtAuthenticationFilter.java`、`JwtUtils.java`
  作用：JWT 生成与解析，认证上下文注入。
  为什么放这一层：属于安全基础设施。
  上下层关系：`SecurityConfig` 把过滤器挂到过滤器链里；Controller 通过 `Authentication` 间接使用它。
  请求是否参与：登录时生成 token，请求进入时解析 token。

- `service`
  真实文件：`AuthService`、`AiService`、`AnalyticsService`、`LeaderboardService`、`WishService`、`QuestionAccessSupport`、`QuestionBankImportService`、`AdminQuestionBankAuditService`
  作用：保留了一部分传统服务接口/实现，以及跨模块能力。
  为什么放这一层：历史代码和跨模块能力没有完全迁进 `module`。
  上下层关系：有些 `ApplicationService` 继续调用这里的 Service，例如 `AuthApplicationService -> AuthService`、`AiController -> AiService`。
  请求是否参与：会，尤其是认证、AI、排行榜、愿望墙。

- `service/impl`
  真实文件：`AuthServiceImpl`、`AiServiceImpl`、`AnalyticsServiceImpl`、`LeaderboardServiceImpl`、`WishServiceImpl`、`AiLimitServiceImpl`
  作用：传统 service 实现层。
  为什么放这一层：这是旧的三层结构风格残留，但现在仍然是核心实现。
  上下层关系：接口在 `service`，实现放 `impl`。
  请求是否参与：会，而且很多重点面试链路还在这里。

- `task`
  真实文件：`WishConsumerTask.java`
  作用：定时消费愿望审核队列。
  为什么放这一层：定时任务不属于 HTTP 接口层。
  上下层关系：依赖 Redis 队列、`WishService`、`AiModelGateway`。
  请求是否参与：不在同步 HTTP 请求里参与，但会影响愿望墙最终可见数据。

### 1.3 module 业务子目录

- `module/auth`
  真实文件：`AuthApplicationService`、`EmailService`、`AuthRateLimitSupport`、`AuthTokenStateSupport`、`AuthPasswordSupport`、`AuthCheckinSupport`、`TurnstileVerifier`、`AvatarSupport`
  作用：登录注册、邮箱验证码、密码重置、签到、头像、风控。
  请求参与：`/api/auth/*` 全链路主模块。

- `module/question`
  真实文件：`QuestionApplicationService`、`QuestionViewAssembler`、`Question*VO`
  作用：题目列表、详情、分类、随机抽题、答案、浏览量。
  请求参与：`/api/questions/*`。

- `module/questionbank`
  真实文件：`QuestionBankApplicationService`、`QuestionBankSupport`、`QuestionBankViewAssembler`
  作用：用户自定义题库上传、查看、重命名。
  请求参与：`/api/question-banks/*`。

- `module/note`
  真实文件：`NoteApplicationService`、`NoteViewAssembler`、`NoteCreateRequest`、`NoteUpdateRequest`
  作用：笔记列表、我的笔记、详情、点赞、评论、创建修改删除。
  请求参与：`/api/notes/*`。

- `module/ai`
  真实文件：`AiModelGateway`、`AiPromptFactory`、`AiCoachSessionSupport`、`AiExternalContextService`、`AiNoteWorkflowSupport`、`AiCallAuditService`
  作用：AI 调用网关、Prompt 组装、SSE 流式输出、会话历史、联网补充上下文、摘要/审核。
  请求参与：`/api/ai/*`。

- `module/analytics`
  真实文件：`AnalyticsApplicationService`、`AnalyticsGeoService`、`AnalyticsVisitStore`
  作用：访客埋点、地理解析、访客画像、仪表盘访客统计。
  请求参与：`/api/analytics/visit`，以及管理端访客统计。

- `module/admin/*`
  真实文件：`dashboard`、`note`、`question`、`questionbank`、`user`、`wish`
  作用：管理后台所有接口。
  请求参与：`/api/admin/**`，统一受管理员权限保护。

- `module/quota`
  真实文件：`AiQuotaPolicy`、`AiQuotaUsageSupport`
  作用：AI 配额、冷却、系统降级。
  请求参与：AI 接口调用前参与。

- `module/system/security`
  真实文件：`CurrentUserService`、`CurrentUser`、`ActorIdentity`
  作用：把 Spring Security 的 `Authentication` 转成业务能理解的用户身份对象。
  请求参与：所有需要区分匿名/游客/注册用户/管理员的接口都会参与。

- `module/system/web`
  真实文件：`ClientRequestService`
  作用：从请求头中提取真实 IP。
  请求参与：登录、注册、验证码发送、访客统计等链路参与。

- `module/system/catalog`
  真实文件：`QuestionCategoryCatalog`
  作用：题目分类标准化和别名扩展。
  请求参与：题目列表筛选、分类下拉、题库分类标准化。

## 2. 项目模块划分

### 2.1 认证鉴权模块

- 模块名称：认证与用户资料模块
- 核心接口：`/api/auth/login`、`/api/auth/register`、`/api/auth/email/send-code`、`/api/auth/password/reset`、`/api/auth/profile`、`/api/auth/streak`
- 核心类：`AuthController`、`AuthApplicationService`、`AuthServiceImpl`、`EmailService`、`JwtUtils`、`JwtAuthenticationFilter`
- 数据流：前端请求 -> `AuthController` -> `AuthApplicationService` -> `AuthServiceImpl` / `EmailService` -> `UserMapper` / Redis -> `JwtUtils` -> 返回 token 和用户信息
- 面试描述：这是一个“密码登录 + 邮箱验证码登录 + 游客登录 + JWT 鉴权 + Redis 限流/验证码/签到”的认证中心。

### 2.2 题库与题目模块

- 模块名称：官方题库与用户题库模块
- 核心接口：`/api/questions/*`、`/api/question-banks/*`、管理端题库审核接口
- 核心类：`QuestionController`、`QuestionApplicationService`、`QuestionAccessSupport`、`QuestionBankController`、`QuestionBankApplicationService`、`AdminQuestionBankAuditService`
- 数据流：请求参数 -> 题库访问校验 -> `QuestionMapper` / `CustomQuestionBankMapper` -> Assembler/VO -> 返回题目或题库数据
- 面试描述：项目把官方题库和用户自定义题库做了统一访问层，但通过 `sourceType/customBankId` 和 `QuestionAccessSupport` 区分可见范围。

### 2.3 笔记模块

- 模块名称：笔记与互动模块
- 核心接口：`/api/notes`、`/api/notes/mine`、`/api/notes/{id}`、点赞评论接口
- 核心类：`NoteController`、`NoteApplicationService`、`NoteMapper`、`NoteLikeMapper`、`NoteCommentMapper`、`NoteViewAssembler`
- 数据流：Controller 接收 DTO -> Service 校验当前用户 -> 操作 `notes/note_likes/note_comments` -> 组装 VO
- 面试描述：这是一个“内容实体 + 互动子表 + 聚合字段实时回填”的典型内容模块。

### 2.4 愿望墙模块

- 模块名称：愿望墙与异步审核模块
- 核心接口：`/api/wishes`、`/api/wishes/{id}/like`、`/api/wishes/{id}/comments`、管理端愿望队列监控与重试
- 核心类：`WishWallController`、`WishServiceImpl`、`WishConsumerTask`、`AdminWishApplicationService`
- 数据流：用户提交愿望 -> `wishes` 表写入待审核记录 -> Redis 待处理队列 -> `WishConsumerTask` 定时消费 -> AI 生成情绪/颜色/浮动速度 -> 更新状态为通过 -> 前台列表展示
- 面试描述：这是项目里最有亮点的异步链路，结合了 MySQL、Redis List/Set、定时任务和 AI 审核。

### 2.5 AI 模块

- 模块名称：AI 面试教练与题解模块
- 核心接口：`/api/ai/question/{id}/explain`、`/api/ai/coach/chat`、`/api/ai/coach/history`、`/api/ai/resume/analyze`、`/api/ai/notes/summarize`
- 核心类：`AiController`、`AiServiceImpl`、`AiModelGateway`、`AiPromptFactory`、`AiCoachSessionSupport`、`AiQuotaUsageSupport`
- 数据流：Controller 收到请求 -> 限流与额度校验 -> Prompt 生成 -> `ChatClient` 调模型 -> SSE 推流或普通返回 -> Redis 记历史/额度 -> 审计日志记录
- 面试描述：AI 不是“直接调接口”，而是封成网关、Prompt 工厂、会话、配额和联网补充上下文几个层次。

### 2.6 排行榜模块

- 模块名称：学习排行榜模块
- 核心接口：`/api/leaderboard`、`/api/leaderboard/question-done`、`/api/leaderboard/game-score`
- 核心类：`LeaderboardController`、`LeaderboardServiceImpl`、`UserQuestionMasteryMapper`
- 数据流：行为上报 -> MySQL 去重记录 / Redis ZSet 排名 -> 聚合排行榜返回
- 面试描述：这个模块把“持久化去重”和“高频排行读取”拆成了 MySQL + Redis 的组合。

### 2.7 访客分析模块

- 模块名称：访客埋点与运营分析模块
- 核心接口：`/api/analytics/visit`、管理端 `/api/admin/visitor-records`
- 核心类：`AnalyticsApplicationService`、`AnalyticsServiceImpl`、`AnalyticsVisitStore`、`AnalyticsGeoService`
- 数据流：访客埋点 -> PV/UV 统计 -> 地理位置解析 -> Redis Hash/ZSet/List/HyperLogLog 形成画像 -> 管理端分页读取
- 面试描述：不是只记一条日志，而是拆成 PV、UV、地理热度、最近访问、访客画像多个维度。

### 2.8 管理后台模块

- 模块名称：后台管理模块
- 核心接口：`/api/admin/**`
- 核心类：`Admin*Controller` + 各自 `ApplicationService`
- 数据流：管理员请求 -> Spring Security `hasRole("ADMIN")` -> 管理 Service -> Mapper -> VO 返回
- 面试描述：管理端和前台业务共享实体层，但通过独立 Controller/VO/DTO 分开，避免前后台接口混在一起。

# 第二部分：技术栈全盘扫描（必须非常详细）

## Spring Boot

- 在项目中的作用：提供自动装配、配置加载、Bean 扫描、Web 容器启动。
- 真实位置：`pom.xml` 使用 `spring-boot-starter-parent:3.2.5`；启动类是 `NovaLeapApplication.java`。
- 参与时机：应用启动时完成容器装配；请求期所有 Bean 都依赖它。
- 为什么用它：省去 XML 配置和样板代码。
- 不用它怎么做：要手工配置 Servlet、Spring MVC、数据源、安全、JSON 转换。
- 面试说法：项目基于 Spring Boot 3，所有 Web、安全、Redis、AI、校验能力都通过 Boot 自动装配拉起来。
- 优化点：目前配置项较多，建议再细化 `@ConfigurationProperties`，减少散落的 `@Value`。

## Spring MVC

- 作用：处理 `@RestController`、参数绑定、JSON 序列化。
- 真实位置：15 个 Controller 全部基于 Spring MVC 注解，例如 `AuthController`、`QuestionController`、`NoteController`。
- 参与时机：前端发请求后先进入 Spring MVC，再路由到具体方法。
- 为什么用它：注解式开发快，参数绑定方便。
- 不用它怎么做：自己写 Servlet + 手工解析 JSON。
- 面试说法：我的接口层是标准 Spring MVC，路径通过 `@RequestMapping/@GetMapping/@PostMapping` 定义，Body 参数通过 DTO 接收，再配合 `@Valid` 做校验。
- 优化点：目前没有统一的 OpenAPI 文档层。

## Spring Security

- 作用：做认证与权限控制。
- 真实位置：`config/SecurityConfig.java`、`security/JwtAuthenticationFilter.java`。
- 参与时机：每次请求进入 Controller 之前先经过 Security Filter Chain。
- 为什么用它：能把匿名、游客、普通用户、管理员分开控制。
- 原生做法：自己写 Filter 判断 token 和权限。
- 面试说法：项目采用 Spring Security 无状态方案，关闭 Session，接入 JWT 过滤器，对 `/api/admin/**` 做管理员角色控制。
- 优化点：权限规则现在都写在 `SecurityConfig`，后续可拆成更细粒度权限策略。

## JWT（jjwt）

- 作用：登录成功后发 token；后续请求靠 token 恢复用户身份。
- 真实位置：`JwtUtils.generateToken/parseToken`、`JwtAuthenticationFilter.doFilterInternal`、`AuthServiceImpl.buildAuthResult`。
- 参与时机：登录/注册/游客登录时生成；请求带 `Authorization: Bearer xxx` 时解析。
- 为什么用它：适合前后端分离，无需服务端 Session。
- 原生做法：使用 Session/Cookie 保存登录态。
- 面试说法：登录后我会在 `AuthServiceImpl.buildAuthResult` 里调用 `JwtUtils.generateToken` 生成带 `subject=username` 和 `role` 的 token，请求进来由 `JwtAuthenticationFilter` 解析后放入 `SecurityContextHolder`。
- 优化点：目前 `logout` 没有真正失效 token，只在重置密码时通过 `AuthTokenStateSupport.invalidateUserTokens` 做失效时间控制。

## MyBatis-Plus

- 作用：简化 CRUD、分页、条件构造。
- 真实位置：所有 Mapper 都继承 `BaseMapper`；`MybatisPlusConfig` 配了 `PaginationInnerInterceptor`。
- 参与时机：Service 层访问数据库时参与。
- 为什么用它：比手写 XML 更快，支持 `LambdaQueryWrapper`、`Page`。
- 原生做法：MyBatis XML / JdbcTemplate 手写 SQL。
- 面试说法：项目数据库层主要用 MyBatis-Plus，像 `NoteApplicationService.getNoteList`、`QuestionApplicationService.getQuestionList`、`AdminUserApplicationService.getUserList` 都是 `Page + LambdaQueryWrapper` 的典型写法。
- 优化点：`application.yml` 配了 `mapper-locations`，但项目几乎没有 XML Mapper，说明配置是冗余的。

## MySQL

- 作用：持久化用户、题目、笔记、愿望、题库、评论、点赞、掌握度。
- 真实位置：`application-dev.yml` 数据源；Entity 对应表如 `users/questions/notes/wishes/custom_question_banks`。
- 参与时机：大多数业务写入和查询最终都落 MySQL。
- 为什么用它：结构化业务数据适合关系型数据库。
- 不用它怎么做：全放 Redis 会丢失持久性和关系查询能力。
- 面试说法：MySQL 承担最终事实数据，Redis 更多承担高频状态和统计。
- 优化点：一些统计型查询可继续做索引优化和读写分离。

## Redis

- 作用：不是单一缓存，而是状态中心。
- 真实位置：`EmailService`、`AuthRateLimitSupport`、`AuthTokenStateSupport`、`AuthCheckinSupport`、`AvatarSupport`、`AnalyticsServiceImpl`、`AnalyticsVisitStore`、`AiQuotaUsageSupport`、`AiCoachSessionSupport`、`AiCallAuditService`、`LeaderboardServiceImpl`、`WishConsumerTask`、`AdminWishApplicationService`、`WishServiceImpl`。
- 参与时机：登录风控、验证码、签到、AI 配额、排行榜、访客分析、愿望队列都会参与。
- 为什么用它：这些场景都需要高频读写和轻量状态存储。
- 原生做法：全放 MySQL 会增加热点写入压力，而且不适合 List/ZSet/Bitmap/HyperLogLog 这种结构。
- 面试说法：我项目里 Redis 用得比较重，典型结构有 String、Hash、List、Set、ZSet、Bitmap、HyperLogLog，不只是缓存。
- 优化点：缺少统一 key 规范文档，后续可以集中管理 key 前缀与 TTL。

## 参数校验（Validation）

- 作用：在 Controller 入参阶段提前拦截非法请求。
- 真实位置：`spring-boot-starter-validation`；DTO 上的 `@NotBlank/@Size/@Email/@Min/@Max`；Controller 上的 `@Valid/@Validated`。
- 参与时机：进入 Controller 方法前。
- 为什么用它：避免业务层重复写 null/长度校验。
- 原生做法：Controller 或 Service 里手写 if 判断。
- 面试说法：请求体统一用 DTO 承接，并配合 `@Valid` 让字段级规则先执行，再进入 Service。
- 优化点：目前部分简单 `Map<String,String>` 请求如发邮箱验证码没有单独 DTO，企业里会补齐。

## 全局异常处理

- 作用：统一把异常转成前端可识别的 JSON。
- 真实位置：`common/GlobalExceptionHandler.java`。
- 参与时机：Controller/Service 抛出异常后。
- 为什么用它：避免每个接口手工 try-catch。
- 原生做法：每个 Controller 自己包装异常。
- 面试说法：项目里自定义了 `BusinessException/UnauthorizedException/ForbiddenException/NotFoundException`，再由 `@RestControllerAdvice` 统一转为 `Result.error(...)`。
- 优化点：现在运行时异常统一返回 500 文案，缺少 traceId 和更细粒度日志上下文。

## 统一返回结果 Result

- 作用：保证前端拿到一致的 `code/msg/data` 结构。
- 真实位置：`common/Result.java`。
- 参与时机：Controller 返回时。
- 为什么用它：前端处理简单，异常与成功格式统一。
- 不用它怎么做：每个接口返回不同结构，前端判断更乱。
- 面试说法：我的 Controller 基本都返回 `Result<T>`，前端只看 `code/msg/data` 三段即可。
- 优化点：企业里通常会补充 traceId、timestamp、errorDetails。

## 事务

- 作用：保证多步写操作一致性。
- 真实位置：`NoteApplicationService`、`WishServiceImpl`、`AdminQuestionBankAuditService`、`AdminNoteApplicationService` 等多个方法标了 `@Transactional`。
- 参与时机：写数据库的复合操作时。
- 为什么用它：避免部分成功部分失败。
- 面试说法：像愿望提交、题库审核导入、笔记创建更新等修改型接口都放在事务里。
- 优化点：目前不是所有需要保护的写链路都加了事务，例如部分点赞评论可继续收紧。

## 分页插件

- 作用：为 `Page<T>` 查询自动拼接分页 SQL。
- 真实位置：`MybatisPlusConfig.mybatisPlusInterceptor()`。
- 参与时机：`selectPage` 调用时。
- 为什么用它：分页查询是后台列表和内容列表的核心需求。
- 面试说法：分页接口统一使用 MyBatis-Plus 的 `Page` 对象配合分页插件。
- 优化点：前台和后台分页参数上限控制还可以更统一。

## Lombok

- 作用：减少 getter/setter/构造器样板代码。
- 真实位置：Entity、DTO、VO 大量使用 `@Data`，少量 `@AllArgsConstructor`。
- 面试说法：项目大量对象类都用 Lombok 简化代码。
- 优化点：企业里部分实体会避免滥用 `@Data`，改为更精确注解。

## Jackson

- 作用：JSON 序列化、反序列化、SSE 消息封装。
- 真实位置：Spring MVC 默认使用；显式出现在 `SecurityConfig`、`AiModelGateway`、`AiCoachSessionSupport`、`AnalyticsVisitStore`、`AiExternalContextService`。
- 面试说法：除了常规 JSON 转换，我还用 `ObjectMapper` 处理 SSE payload、Redis 中的会话历史和访客记录 JSON。
- 优化点：当前没有统一 Jackson 配置类。

## Spring AI

- 作用：通过 `ChatClient` 调兼容 OpenAI 的模型。
- 真实位置：`pom.xml` 引入 `spring-ai-openai-spring-boot-starter`，`AiModelGateway` 注入 `ChatClient.Builder`。
- 参与时机：AI 题解、教练聊天、简历分析、笔记摘要、愿望情绪识别。
- 为什么用它：比手写 HTTP 更方便统一模型调用。
- 不用它怎么做：手写 HTTP 请求拼 OpenAI 协议。
- 面试说法：AI 模块底层不是直接调某个厂商 SDK，而是通过 Spring AI 的 `ChatClient` 封成统一网关，方便后面换模型。
- 优化点：当前使用的是 Milestone 版本，稳定性和升级风险要关注。

## SSE（Server-Sent Events）

- 作用：流式返回 AI 结果。
- 真实位置：`AiController` 返回 `SseEmitter`；`AiModelGateway.streamModelAnswer/streamStaticAnswer`。
- 参与时机：AI 题解、AI 教练、简历分析。
- 为什么用它：前端能边生成边显示。
- 原生做法：同步等待整段文本生成后一次返回。
- 面试说法：我项目里 AI 输出走的是 SSE，`AiModelGateway` 会把模型 delta 包装成 `type=delta/done` 的消息推给前端。
- 优化点：可以继续补心跳、断线重连、取消生成。

## JDK HttpClient

- 作用：调用 Cloudflare Turnstile、IP 定位、联网搜索、天气接口。
- 真实位置：`TurnstileVerifier`、`AnalyticsGeoService`、`AiExternalContextService`。
- 为什么用它：JDK 17 自带，无需额外依赖。
- 说明：项目没有用 `RestTemplate/WebClient/OkHttp`。
- 面试说法：第三方 HTTP 调用统一采用 Java 17 的 `HttpClient`，场景包括验证码风控校验、IP 地理位置解析、AI 外部上下文补充。

## Resend 邮件服务

- 作用：发送邮箱验证码。
- 真实位置：`module/auth/service/EmailService.java`，依赖是 `com.resend:resend-java`。
- 参与时机：注册、登录、重置密码发送邮箱验证码时。
- 为什么用它：比自建 SMTP 更轻。
- 优化点：`pom.xml` 使用了 `LATEST`，这在企业项目里不建议。

## Cloudflare Turnstile

- 作用：人机校验，防刷注册/登录。
- 真实位置：`TurnstileVerifier.verifyIfEnabled`。
- 参与时机：登录、注册前。
- 为什么用它：减少恶意脚本刷接口。
- 说明：这是“代码接通但按配置可关闭”的技术点，`nova.turnstile.enabled` 为 false 时不参与。

## 定时任务

- 作用：异步消费愿望审核队列、恢复卡住任务、补偿漏入队任务。
- 真实位置：`task/WishConsumerTask.java`，使用 `@EnableScheduling + @Scheduled`。
- 参与时机：后台周期性执行，不依赖 HTTP 请求。
- 面试说法：愿望审核不是同步做的，而是提交后入 Redis，定时任务批量消费，提高提交接口响应速度。

## Docker / Docker Compose

- 作用：容器化部署 MySQL、Redis、后端、前端、管理端。
- 真实位置：`nova-backend/Dockerfile`、根目录 `docker-compose.yml`。
- 说明：项目已经具备基础容器化能力。

## 逻辑删除

- 作用：删除评论、笔记时保留数据痕迹。
- 真实位置：`Note.deleted`、`NoteComment.deleted`、`WishComment.deleted` 使用 `@TableLogic`；`application.yml` 配了全局逻辑删除字段。
- 面试说法：项目部分内容表采用逻辑删除，避免直接物理删除导致数据不可追溯。

## 明确“没有”或“未落地”的技术

- 没有 `Swagger/Knife4j/OpenAPI`，源码和依赖里都没接。
- 没有 `WebMvcConfigurer`，也没有显式 CORS 配置。
- 没有 Spring MVC `HandlerInterceptor`，真正参与请求的是 Spring Security Filter。
- 没有 AOP/`@Aspect`。
- 没有 `ThreadLocal` 形式的用户上下文透传，用户信息主要靠 `Authentication` 和 `CurrentUserService`。
- 没有 `langchain4j`。
- 没有 OSS/对象存储；所谓上传只是 `MultipartFile` 读取 TXT 内容，没有落盘、没有云存储。
- 没有通用幂等组件；只有局部幂等处理，比如 `LeaderboardServiceImpl.recordQuestionDone` 用唯一约束+`DuplicateKeyException` 保证重复点击安全。
- 没有 Spring Cache 注解体系，Redis 主要被当作业务状态库使用。

# 第三部分：技术栈 -> 文件 -> 模块 -> 接口 映射（必须细）

## JWT 鉴权链

- 生成位置：`AuthServiceImpl.buildAuthResult()` 调 `JwtUtils.generateToken(username, role)`。
- 返回位置：`AuthController.login/register/guestLogin` 最终把 token 放进 `Result.success(Map<String,Object>)`。
- 解析位置：`JwtAuthenticationFilter.doFilterInternal()` 从 `Authorization` 头里截出 `Bearer ` 后的 token。
- 失效控制：`AuthTokenStateSupport.isTokenActive(username, issuedAt)`；重置密码时 `invalidateUserTokens(username)`。
- 用户信息落点：`SecurityContextHolder.getContext().setAuthentication(...)`。
- Controller 读取方式：Controller 直接接 `Authentication` 参数；再由 `CurrentUserService.current/requireDatabaseUser/resolveActor` 转业务身份。
- 依赖接口：除了 permitAll 的公开接口，绝大部分接口都依赖它，特别是 `/api/auth/profile`、`/api/notes/*`、`/api/question-banks/*`、`/api/ai/*`、`/api/admin/**`。

## Redis 验证码链

- 生成位置：`EmailService.sendVerificationCode()`。
- 存储位置：`verify_code:type:email`、冷却 key、IP 冷却 key。
- 校验位置：`AuthApplicationService.login/register/resetPassword` 调 `emailService.verifyCode(...)`。
- 销毁位置：成功后 `emailService.consumeCode(...)`。
- 依赖接口：`POST /api/auth/email/send-code`、`POST /api/auth/register`、`POST /api/auth/login`（验证码登录）、`POST /api/auth/password/reset`。

## Redis 登录风控链

- 位置：`AuthRateLimitSupport`。
- 参与方法：`checkLoginRateLimit`、`recordLoginFailure`、`isLoginLocked`、`clearLoginFailures`、`checkRegisterRateLimit`。
- 依赖接口：登录、注册。
- 作用：限制单位时间请求次数，限制同 IP + 用户名连续失败次数。

## MyBatis-Plus 分页链

- 配置位置：`MybatisPlusConfig`。
- 典型接口：
  - `QuestionApplicationService.getQuestionList`
  - `NoteApplicationService.getNoteList/getMine`
  - `QuestionBankApplicationService.getMyBanks`
  - 全部 `Admin*ApplicationService.get*List`
- 典型方法：`new Page<>(page,size)` + `mapper.selectPage(pageParam, wrapper)`。
- 作用：自动把分页参数转成 SQL limit。

## 题库访问控制链

- 核心类：`QuestionAccessSupport`。
- 方法：`resolveAccessibleBank`、`resolveAccessibleQuestion`。
- 参与接口：`GET /api/questions`、`GET /api/questions/{id}`、`GET /api/questions/random`、`GET /api/questions/{id}/answer`、`GET /api/ai/question/{id}/explain`。
- 作用：如果题目来自自定义题库，则校验当前用户是不是题库拥有者或管理员。

## AI 调用链

- Controller：`AiController`。
- 业务入口：`AiServiceImpl`。
- Prompt 组装：`AiPromptFactory`。
- 模型网关：`AiModelGateway` 使用 `ChatClient`。
- 限流与配额：`AiLimitServiceImpl` -> `AiQuotaPolicy` + `AiQuotaUsageSupport`。
- 会话历史：`AiCoachSessionSupport` 存 Redis List。
- 联网补充上下文：`AiExternalContextService`。
- 审计：`AiCallAuditService`。
- 接口落点：题解、教练、简历分析、笔记摘要、愿望情绪分析。

## 愿望墙异步队列链

- 提交：`WishServiceImpl.submit`。
- 入队：`enqueuePendingWish` 把 `wishId` 放进 `wish:pending`。
- 消费：`WishConsumerTask.processPendingWishes`。
- 处理中队列：`wish:processing`。
- 分布式锁：`wish:processing:lock:{id}`。
- 重试计数：`wish:retry:{id}`。
- 死信集合：`wish:dead`。
- 管理端观测：`AdminWishApplicationService.getWishQueueStats/getDeadLetterWishes/retryWishReview`。

## 排行榜链

- 做题完成：`LeaderboardController.markQuestionDone` -> `LeaderboardServiceImpl.recordQuestionDone` -> `user_question_mastery` + Redis Set/ZSet。
- 游戏得分：`recordGameScore` -> Redis ZSet `nova:leaderboard:game_best`。
- 发愿望积分：`WishServiceImpl.submit` 提交成功后 `leaderboardService.recordWishPost(username)`。
- 榜单读取：`LeaderboardServiceImpl.getLeaderboard` 聚合 MySQL + Redis。

## 访客分析链

- 请求入口：`POST /api/analytics/visit`。
- 应用服务：`AnalyticsApplicationService.trackVisit`。
- 核心服务：`AnalyticsServiceImpl.trackVisit`。
- Redis 结构：String(PV)、HyperLogLog(UV)、Hash(访客画像)、ZSet(最近访问排序/地理热度)、List(最近地理访问)。
- 地理解析：`AnalyticsGeoService.resolveGeoInfo`。
- 管理端读取：`AdminDashboardApplicationService.getVisitorRecords` -> `AnalyticsService.getVisitorRecords`。

# 第五部分：所有接口的实现分析（逐个解释）

说明：这一部分我不再只做“接口索引”，而是按接口逐个给你讲清楚 Controller、Service、Mapper、对象、技术、执行流程、面试说法和优化点。为了覆盖全部接口，我采用“每个接口一段精炼全链路说明”的方式。

## 5.1 认证模块 `AuthController`

### `POST /api/auth/login` -> `AuthController.login`

- 作用：解决用户登录问题，支持密码登录和邮箱验证码登录两种模式，属于整个系统的认证入口。Controller 用 `@PostMapping` 接 `LoginRequest`，参数来自 JSON Body，`@Valid` 先做基础校验。真正业务在 `AuthApplicationService.login`，它先通过 `ClientRequestService.resolveClientIp` 取 IP，再根据 `request.isCodeLogin()` 决定走 `EmailService.verifyCode + AuthServiceImpl.loginByEmailCode` 还是 `AuthServiceImpl.login`。`AuthServiceImpl.login` 会做 Redis 登录限流、失败锁定、Turnstile 校验、`UserMapper` 查库、`AuthPasswordSupport.verifyPasswordAndMigrateIfLegacy` 校验密码，最后在 `buildAuthResult` 里调用 `JwtUtils.generateToken` 生成 token 并返回。对象上 DTO 是 `LoginRequest`，返回故意不用 Entity 而是 `Map<String,Object>` 包 `token/user/avatar/checkin`。技术点是 Validation、Redis、JWT、BCrypt、Spring Security、Turnstile。完整流程：前端提交账号密码/验证码 -> Controller 绑定 DTO -> 参数校验 -> `AuthApplicationService` 分流 -> `AuthServiceImpl` 查 MySQL/校验 Redis 风控 -> 生成 JWT -> `Result.success(...)` 返回。面试可以说：我把登录拆成应用层和认证服务层，应用层处理登录方式分流，认证服务层处理密码校验、风控和 token 生成。优化点：返回值最好定义专门 VO；错误文案和审计日志可更细；`logout` 还没配套 token 失效。

### `POST /api/auth/guest` -> `AuthController.guestLogin`

- 作用：给游客直接下发 GUEST 身份 token。Controller 很薄，只转发到 `AuthApplicationService.guestLogin`。核心逻辑在 `AuthServiceImpl.guestLogin`：构造一个不落库的 `User` 对象，用户名形如 `guest-uuid`，角色写死 `GUEST`，然后复用 `buildAuthResult` 生成 token 和安全用户对象。Mapper 不查库。技术点是 JWT 和角色区分。流程：前端请求 -> Controller -> `guestLogin` -> 构造游客 User -> `JwtUtils.generateToken` -> 返回。面试说法：游客模式没有真正写数据库，而是生成一个带 GUEST 角色的 JWT，让权限链统一处理。优化点：游客标识前缀和其他类里的判断并不完全一致，后面我会在问题排查里展开。

### `POST /api/auth/register` -> `AuthController.register`

- 作用：新用户注册并直接返回登录态。Controller 用 `RegisterRequest` 接 Body。`AuthApplicationService.register` 先校验邮箱验证码，再把 IP 和业务参数传给 `AuthServiceImpl.register`。`AuthServiceImpl.register` 会做邮箱标准化、密码强度校验、确认密码校验、同意协议校验、Redis 注册限流、Turnstile 校验、`UserMapper` 唯一性检查，最后保存用户并调用 `buildAuthResult` 返回 token。DTO 是 `RegisterRequest`，Entity 是 `User`，返回仍然是 `Map<String,Object>`。技术点：Validation、邮箱验证码、Redis、BCrypt、JWT、DataIntegrityViolationException 兜底并发唯一性。面试说法：注册接口不是单纯 insert，而是加了验证码、风控、密码强度和数据库唯一约束双保险。优化点：最好把返回结构 VO 化，并记录注册审计日志。

### `POST /api/auth/logout` -> `AuthController.logout`

- 作用：名义上退出登录。真实实现只有 `return Result.success()`，没有调任何 Service，也没有清 Redis 黑名单、没有失效 token。这个接口在系统里位置是“占位式接口”。面试时不要说它做了完整注销；应该诚实说当前实现主要依赖前端删 token，本地没有做真正服务端失效。优化建议：调用 `AuthTokenStateSupport.invalidateUserTokens` 或维护 token 黑名单。

### `POST /api/auth/password/reset` -> `AuthController.resetPassword`

- 作用：用邮箱验证码重置密码。Controller 接 `PasswordResetRequest`。`AuthApplicationService.resetPassword` 校验验证码后调用 `AuthServiceImpl.resetPassword`；后者做密码强度校验、`UserMapper` 查用户、BCrypt 重编码，并通过 `AuthTokenStateSupport.invalidateUserTokens` 让旧 token 基于签发时间失效。流程里 Redis 既参与验证码，也参与 token 失效时间。面试说法：重置密码除了改数据库，我还做了 token 失效时间控制，避免旧 token 继续使用。优化点：方法参数里的 `clientIp/userAgent` 目前没被真正利用。

### `POST /api/auth/email/send-code` -> `AuthController.sendEmailCode`

- 作用：发送注册/登录/重置验证码。Controller 这里没有专门 DTO，而是直接收 `Map<String,String>`，这是一处工程化不够整齐的地方。`AuthApplicationService.sendEmailCode` 先标准化邮箱和类型，再通过 `AuthService.emailExists` 判定账号状态，最后 `EmailService.sendVerificationCode` 把验证码存 Redis、设置邮箱冷却和 IP 冷却，然后调用 Resend 发邮件。面试说法：这个接口里 Redis 主要不是缓存，而是做验证码 TTL 和防刷冷却。优化点：补专门请求 DTO，避免魔法字符串 key。

### `GET /api/auth/profile` -> `AuthController.getProfile`

- 作用：获取当前登录用户资料。Controller 从 Spring Security 注入 `Authentication`。`AuthApplicationService.getProfile` 先用 `CurrentUserService.requireUsername` 保证已登录，再走 `AuthServiceImpl.getProfile`。如果是游客角色，直接拼游客资料返回；如果是真用户，就用 `UserMapper` 查库，再组装 `id/username/nickname/role/avatar/accountType`。技术点是 JWT 解析后的 `Authentication`、头像 Redis。优化点：返回 `Map` 而不是 `ProfileVO`。

### `PUT /api/auth/profile` -> `AuthController.updateProfile`

- 作用：改昵称、密码、头像。Controller 用 `ProfileUpdateRequest` 接 Body。`AuthApplicationService.updateProfile` 取当前用户，`AuthServiceImpl.updateProfile` 根据字段是否有值做增量修改：昵称直接改，密码先校验强度再 BCrypt 编码，头像交给 `AvatarSupport.normalizeAvatarInput/isAllowedAvatar/saveAvatar` 存 Redis。流程是 Controller -> DTO -> 当前用户 -> `UserMapper.updateById` + Redis 头像。面试说法：资料更新走的是“数据库 + Redis”双写模型，基础资料在 MySQL，头像选择在 Redis。优化点：如果密码更新成功，最好同步让旧 token 失效。

### `GET /api/auth/streak` -> `AuthController.getCheckinStreak`

- 作用：获取签到连续天数。`AuthApplicationService.getCheckinStreak` -> `AuthServiceImpl.getCheckinInfo` -> `AuthCheckinSupport.resolveCheckinPayload`。真实存储不是表，而是 Redis Bitmap，按 `nova:checkin:userId:yyyyMM` 存月度签到位。`resolveCheckinPayload` 会算今天是否已签、连续天数是多少。面试说法：签到用 Bitmap 比单独建签到表更省空间。优化点：当前登录时默认触发签到，这个规则要和产品定义对齐。

## 5.2 题目模块 `QuestionController`

### `GET /api/questions` -> `QuestionController.getQuestionList`

- 作用：分页查题目，支持官方题库和自定义题库。Controller 接 Query 参数。`QuestionApplicationService.getQuestionList` 先 `new Page<>(normalizePage, normalizeSize)`，再用 `LambdaQueryWrapper<Question>` 拼条件：必须 `status=1`，如果传 `bankId` 就先用 `QuestionAccessSupport.resolveAccessibleBank` 校验题库可见性，再限制 `sourceType=CUSTOM and customBankId=bankId`；否则默认只查官方题 `sourceType=OFFICIAL or null`。分类过滤用 `QuestionCategoryCatalog.expandFilterCodes` 做别名展开，最后 `questionMapper.selectPage`。返回前走 `QuestionViewAssembler.toListItemVO`。面试说法：这个列表接口的难点不在分页，而在“题目可见范围”和“分类标准化”。优化点：随机/列表都依赖 wrapper 现场拼装，后续可以抽查询规格类。

### `GET /api/questions/categories` -> `QuestionController.getQuestionCategoryList`

- 作用：给前端下拉框返回分类。核心在 `QuestionApplicationService.loadQuestionCategoryOptions`：优先读 `question_categories` 表；如果 DB 数据为空或异常，则回退到 `QuestionCategoryCatalog` 里的内置分类。面试说法：这是“数据库配置优先，代码内置兜底”的设计。优化点：分类可以做本地缓存，减少每次查表。

### `GET /api/questions/{id}` -> `QuestionController.getQuestionDetail`

- 作用：查看题目详情。Service 先 `QuestionAccessSupport.resolveAccessibleQuestion` 校验题目是否存在、是否上架、如果是自定义题库是否当前用户有权访问，然后 `QuestionViewAssembler.toDetailVO` 返回。Mapper 层是 `questionMapper.selectById` 和可能的 `customQuestionBankMapper.selectById`。面试说法：详情接口不只是查一条题目，还把自定义题库权限判断前置掉了。

### `POST /api/questions/{id}/view` -> `QuestionController.increaseQuestionView`

- 作用：浏览量 +1。`QuestionApplicationService.increaseQuestionView` 先做可访问校验，再用 `UpdateWrapper.setSql("view_count = IFNULL(view_count, 0) + 1")` 做原子更新，最后重新查一遍题目取最新 viewCount。面试说法：我这里没有先查再加再写，而是用 SQL 自增，避免并发覆盖。优化点：现在更新后还查一次，可考虑直接返回增量结果或异步累计。

### `GET /api/questions/random` -> `QuestionController.drawRandomQuestion`

- 作用：随机抽题。Service 会在拼完筛选条件后先 `selectCount` 拿总数，再用 `ThreadLocalRandom` 生成 offset，最后 `last("LIMIT offset,1")` 随机查一条。面试说法：我没有把全部题先查出来再随机，而是 count + offset 方式减少内存开销。优化点：大表下 offset 性能一般，企业里会进一步优化成随机主键或预采样。

### `GET /api/questions/{id}/answer` -> `QuestionController.getQuestionAnswer`

- 作用：返回标准答案。Service 会优先取 `Question.standardAnswer`，为空再退回 `content`，之后调用 `sanitizeOfficialAnswer` 清掉历史导入前缀，再通过 `QuestionViewAssembler.toAnswerVO` 返回。面试说法：答案接口额外做了历史脏数据兼容，避免把“标准答案:”这类前缀直接回给前端。

## 5.3 用户题库模块 `QuestionBankController`

### `GET /api/question-banks/mine` -> `QuestionBankController.getMyBanks`

- 作用：查当前用户上传的题库。`QuestionBankApplicationService.getMyBanks` 先 `CurrentUserService.requireDatabaseUser` 禁止游客，再按 `userId` 和可选 `status` 用 `CustomQuestionBankMapper.selectPage` 分页查询，最后 `QuestionBankViewAssembler.toUserVO` 组装。面试说法：用户题库列表本质是一个 owner-scope 查询，不是全表查。优化点：分页参数没有统一上限校验。

### `POST /api/question-banks/import` -> `QuestionBankController.importQuestionBank`

- 作用：上传 TXT 题库并保存为待审核状态。Controller 用 `MultipartFile` 接文件。`QuestionBankApplicationService.importQuestionBank` 先做大小校验，再调用 `QuestionBankImportService.analyzeFile` 解析出 `ImportPayload(fileType, rawText, questions)`，然后把原始内容和元信息写入 `custom_question_banks`，状态设为 `PENDING`，并不立即导入题目表。面试说法：这里的上传不是文件落盘，而是“解析内容 + 入库等待审核”。优化点：现在只支持 txt，不支持 docx/pdf。

### `PUT /api/question-banks/{id}` -> `QuestionController.renameQuestionBank`

- 作用：重命名题库。Service 先查 `CustomQuestionBank`，再校验是不是拥有者或管理员，然后更新名称和 `updatedAt`。技术点是 JWT、权限控制、MyBatis-Plus。面试说法：这个接口体现了对象级权限，不是登录就能改，必须 owner 或 admin。

## 5.4 笔记模块 `NoteController`

### `GET /api/notes` -> `NoteController.getNoteList`

- 作用：分页查通过审核的笔记列表。`NoteApplicationService.getNoteList` 用 `Page<Note>` + `LambdaQueryWrapper` 查 `status=1` 的笔记，再根据 keyword/category 拼条件。查出后不会直接返回 Entity，而是对每条笔记调用 `enrichNoteMetadata` 动态回填点赞数、评论数、是否我已点赞，再交给 `NoteViewAssembler.toListItemVO`。面试说法：这是内容聚合查询，主体表是 notes，互动数据来自 note_likes/note_comments。优化点：现在 N 条笔记会重复 count，存在 N+1 风险，企业里会改成批量聚合。

### `GET /api/notes/mine` -> `NoteController.getMine`

- 作用：查当前用户自己的笔记。Service 先 `requireDatabaseUser`，然后按 `userId` 查 notes，可选 status 过滤，最后同样走 `enrichNoteMetadata + Assembler`。面试说法：我的“我的笔记”和“公开笔记列表”复用了同一套组装逻辑，只是查询条件不同。

### `GET /api/notes/{id}` -> `NoteController.getNoteDetail`

- 作用：查一篇笔记详情。当前实现是 `noteMapper.selectById(id)` 后直接 enrich 并返回，没有限制公开状态，也没判断是不是作者。这意味着只要已登录且知道 ID，就可能看到待审核/驳回笔记，这是后面需要重点追问的点。面试时可以诚实说当前实现能跑，但企业里应补状态和 ownership 校验。

### `POST /api/notes/{id}/view` -> `NoteController.increaseView`

- 作用：笔记浏览量 +1。逻辑是查 `Note`，如果不存在报 404，然后 `note.setViewCount(...)` 再 `noteMapper.updateById(note)`。这和题目的原子 SQL 自增不同，并发下可能出现覆盖。面试说法可以说：当前是先查再改，后续会改成 SQL 原子自增以提高并发安全。

### `POST /api/notes` -> `NoteController.createNote`

- 作用：创建笔记。Service 必须要求真实用户，构建 `Note` 时 `userId` 用当前用户 id，`author` 强制写当前用户昵称，`status=2` 代表待审核，`createdAt/updatedAt` 为当前时间，然后 `noteMapper.insert(note)`。对象职责上 `NoteCreateRequest` 只负责接参，`Note` 才是数据库对象，`NoteDetailVO` 是返回对象。面试说法：用户提交后不是立刻公开，而是先进入审核状态。优化点：当前没有接入 `AiService.moderateNote` 自动审核，审核链只做到了“状态字段预留”。

### `PUT /api/notes/{id}` -> `NoteController.updateNote`

- 作用：修改自己的笔记。Service 先加载当前用户，再查笔记并校验 `note.userId == currentUser.id`，修改标题/内容/摘要/分类/emoji，最后把 `status` 重置为 2 重新审核。面试说法：更新接口不是直接覆盖发布状态，而是重新进入审核流。优化点：依旧缺少自动审核实际接入。

### `POST /api/notes/{id}/like` -> `NoteController.toggleLike`

- 作用：点赞/取消点赞。Service 用 `CurrentUserService.resolveActor` 同时支持普通用户和游客身份，然后查 `note_likes` 是否已有 `(noteId, actorType, actorId)` 记录，有则删，无则插。最后统计总点赞数封成 `NoteLikeVO`。面试说法：我把点赞主体抽象成 `ActorIdentity`，这样用户和游客能复用一套点赞逻辑。优化点：当前没有校验笔记状态是否公开。

### `GET /api/notes/{id}/comments` -> `NoteController.getComments`

- 作用：查评论列表。Service 直接按 `noteId` 从 `note_comments` 倒序查，再组装 `NoteCommentVO`。这里 `authentication` 参数实际上没有被利用，`mine` 字段也没赋值，是一处实现不完整的地方。面试时如果被追问可以主动指出：接口能跑，但“我的评论标记”没真正做完。

### `POST /api/notes/{id}/comments` -> `NoteController.addComment`

- 作用：给笔记发表评论。Service 只校验了当前用户是否登录且非游客，直接构造 `NoteComment` 插入，没有先校验笔记是否存在、是否已通过审核。流程是 Controller -> DTO -> 当前用户 -> `noteCommentMapper.insert` -> 组装返回。优化点：企业里通常会先校验 note 存在且允许评论。

### `DELETE /api/notes/{id}` -> `NoteController.deleteNote`

- 作用：删除自己的笔记。Service 查笔记并校验归属后 `noteMapper.deleteById(id)`。由于 `Note` 有 `@TableLogic`，这里走的是逻辑删除。面试说法：笔记删除不是物理删，而是逻辑删。
## 5.5 愿望墙模块 `WishWallController`

### `GET /api/wishes` -> `WishWallController.getApprovedWishes`

- 作用：展示愿望墙。Controller 允许匿名带 `visitorId` 访问，也允许登录用户访问。它先用私有方法 `resolveActor(authentication, visitorId)` 组装 `ActorIdentity`，再调用 `WishServiceImpl.listApproved`。Service 层按 `status=APPROVED` 查最近 50 条愿望，然后批量统计点赞数、评论数、当前访客是否已点赞，封成 `WishWallItemVO`。技术点是 MyBatis-Plus + Redis 身份辅助 + 游客/用户双身份。面试说法：这个列表接口的关键不是查 wish 表，而是把互动字段聚合回前台卡片。

### `POST /api/wishes` -> `WishWallController.submitWish`

- 作用：用户发布愿望。Controller 强制要求真实数据库用户，游客不能发。`WishServiceImpl.submit` 在事务里创建一条 `status=PENDING` 的愿望，然后通过 `runAfterCommit` 在事务提交后把 `wishId` 推进 Redis `wish:pending` 队列，并尝试给排行榜记一分。面试说法：我把耗时审核异步化了，提交接口只负责入库和入队，所以响应比较快。优化点：目前没有防重复提交或敏感词同步初筛。

### `POST /api/wishes/{wishId}/like` -> `WishWallController.toggleLike`

- 作用：用户或访客点赞愿望。Controller 对游客账号直接禁掉，但匿名访客可以携带 `visitorId` 点赞。Service 先校验愿望必须已审核通过，再根据 `(wishId, actorType, actorId)` 判断是插入还是删除 `wish_likes`。最后 count 出总点赞数。面试说法：点赞不是只面向登录用户，匿名访客通过 visitorId 也能形成独立身份。优化点：visitorId 可信度依赖前端，企业里会再签名或下发服务端 visitor token。

### `GET /api/wishes/{wishId}/comments` -> `WishWallController.getComments`

- 作用：查看愿望评论。Controller 会把当前用户名传给 Service，用于标记评论是不是自己发的。`WishServiceImpl.listComments` 先确认愿望已审核通过，再按时间倒序查 `wish_comments`，最后 `toCommentVO` 填 `mine` 字段。面试说法：评论列表在 Service 里会带当前用户视角，返回时就知道哪条是我发的。

### `POST /api/wishes/{wishId}/comments` -> `WishWallController.addComment`

- 作用：登录用户评论愿望。Controller 禁止游客，拿到真实 `User` 后把 `userId/username/nickname/content` 全传给 `WishServiceImpl.addComment`。Service 会校验内容长度、愿望状态，再写 `wish_comments` 表。面试说法：评论接口的身份数据不是让前端传，而是后端从当前登录用户里拿，避免伪造用户名。

## 5.6 AI 模块 `AiController`

### `GET /api/ai/question/{id}/explain` -> `AiController.explainQuestion`

- 作用：流式讲解某道题。Controller 先通过 `resolveIdentifier` 把当前请求身份映射成用户名或 IP，再调用 `QuestionAccessSupport.resolveAccessibleQuestion` 做题目访问校验，校验不过就给 `aiService.explainQuestion(identifier, -1L)` 返回静态错误流。`AiServiceImpl.explainQuestion` 会校验 AI 是否可用、额度是否足够，再通过 `AiPromptFactory.buildQuestionExplainPrompt` 组 Prompt，交给 `AiModelGateway.streamModelAnswer` 推 SSE。面试说法：题解接口不是任何题都能讲，先走题目访问控制，再走 AI 配额，再进入模型网关。

### `POST /api/ai/coach/chat` -> `AiController.coachChat`

- 作用：AI 面试教练对话。Controller 接 `AiCoachChatRequest`，会兜底 topic/mode。`AiServiceImpl.coachChat` 先清洗消息，判断是否身份问答/创作者问答，如果是就走固定回复；否则从 `AiCoachSessionSupport.getCoachHistory` 拿最近对话，从 `AiExternalContextService.buildExternalContext` 拿联网补充内容，再由 `AiPromptFactory.buildCoachPrompt` 生成最终 Prompt，最后 `AiModelGateway.streamModelAnswer` 流式返回，并把用户消息和模型回复写回 Redis 会话历史。面试说法：这个接口体现了“Prompt 工厂 + 历史会话 + 外部上下文 + 流式输出 + 额度控制”的完整 AI 服务化封装。

### `GET /api/ai/coach/history` -> `AiController.coachHistory`

- 作用：查当前用户会话历史。Controller 用 `CurrentUserService.requireUsername` 拿当前用户名，然后 `AiServiceImpl.getCoachHistory` 返回 Redis 里的历史记录，再通过 `AiViewAssembler.toCoachHistory` 转 VO。面试说法：会话历史没落 MySQL，而是放 Redis List，适合短期聊天记录。

### `POST /api/ai/coach/session/new` -> `AiController.createCoachSession`

- 作用：新建聊天 session。Service 其实就是 `AiCoachSessionSupport.createCoachSession` 往 Redis Value 写一个新的 `sess_xxx`。这个接口的意义是把多轮聊天隔开。面试说法：聊天历史不只是按用户区分，还按 sessionId 分段。

### `DELETE /api/ai/coach/history` -> `AiController.clearCoachHistory`

- 作用：清空当前用户历史。Service 直接删 Redis 中的 history key 和 active session key。面试说法：清理动作是逻辑删除 Redis 会话，不会去删数据库，因为本来就没落 MySQL。

### `POST /api/ai/resume/analyze` -> `AiController.analyzeResume`

- 作用：流式简历分析。流程和题解类似：参数校验 -> 额度校验 -> `AiPromptFactory.buildResumePrompt` -> `AiModelGateway.streamModelAnswer`。区别在于模块类型是 `AiModule.RESUME`，额度和 maxTokens 会更大。面试说法：简历分析走独立模块配额，不和教练聊天完全共用。

### `POST /api/ai/notes/summarize` -> `AiController.summarizeNote`

- 作用：生成笔记三点摘要。`AiServiceImpl.summarizeNote` 最终转到 `AiNoteWorkflowSupport.summarizeNote`，如果 AI 不可用或额度不足，就走 `AiContentSupport.summarizeByRule` 的规则摘要；可用时就用 `AiPromptFactory.buildNoteSummaryPrompt` 调模型，再把结果解析成 3 个点。面试说法：这个接口做了 AI 能力降级，不是模型挂了就整个接口报错。

## 5.7 排行榜模块 `LeaderboardController`

### `GET /api/leaderboard` -> `LeaderboardController.getLeaderboard`

- 作用：获取总榜。`LeaderboardServiceImpl.getLeaderboard` 先查 `users` 表里的 USER 角色，再从 `user_question_mastery` 聚合做题数，同时从 Redis ZSet 取游戏分数和愿望积分，最后算 `totalScore = questionDone*20 + wishCount*16 + gameBest` 并排序。面试说法：榜单读取是“数据库事实 + Redis高频统计”的混合聚合。

### `GET /api/leaderboard/question-done` -> `LeaderboardController.getDoneQuestions`

- 作用：返回当前用户已完成题目 ID 列表。匿名和游客直接返回空数组；普通用户则 `LeaderboardServiceImpl.getDoneQuestionIds` 同时读 MySQL `user_question_mastery` 和 Redis Set `nova:leaderboard:qset:username`，再去重合并。面试说法：这一步用了 DB + Redis 双读，保证数据恢复能力和读取性能。

### `POST /api/leaderboard/question-done` -> `LeaderboardController.markQuestionDone`

- 作用：上报做完一道题。Controller 强制真实用户。`LeaderboardServiceImpl.recordQuestionDone` 会先校验用户和题目是否存在、是否已做过，再往 `user_question_mastery` 插一条记录；如果并发下重复插入，靠唯一约束和 `DuplicateKeyException` 兜底保证幂等。之后把结果同步到 Redis Set/ZSet。面试说法：这条链路的重点是幂等，避免重复点击把积分刷高。

### `POST /api/leaderboard/game-score` -> `LeaderboardController.reportGameScore`

- 作用：上报游戏分。Service 只认真实用户，然后更新 Redis ZSet `nova:leaderboard:game_best`，仅当本次分数更高时才覆盖。面试说法：高频排行榜写入不走 MySQL，直接落 Redis ZSet。

## 5.8 访客分析模块 `AnalyticsController`

### `POST /api/analytics/visit` -> `AnalyticsController.trackVisit`

- 作用：埋点一次访问。Controller 接 `VisitTrackRequest` 和 `Authentication/HttpServletRequest`。`AnalyticsApplicationService.trackVisit` 会区分 actorType：匿名访客是 guest，游客账号还是 guest，注册用户才是 user；然后把 `path/ip/visitorId/actorName` 交给 `AnalyticsServiceImpl.trackVisit`。后者会更新 PV String、UV HyperLogLog、路径 PV、地理统计 ZSet，并借助 `AnalyticsGeoService.resolveGeoInfo` 和 `AnalyticsVisitStore.recordVisitor` 形成访客画像。面试说法：这个接口不是简单日志，而是实时运营统计入口。优化点：现在所有分析都在 Redis，长期归档方案还没有。

- 作用：审核用户题库。Service 先查 bank，标准化状态/分类/难度/驳回原因；如果审核通过且未导入过，就执行 `importQuestions(bank)`，把 rawContent 解析成多个 `Question` 并写入 `questions` 表，`sourceType=CUSTOM`，同时回写 `importedQuestionCount/importedAt`。面试说法：题库审核接口的核心不是改个 status，而是“审核通过时触发题目批量入库”。

### `GET /api/admin/users` -> `AdminUserController.getUserList`

- 作用：后台用户列表。Service 用 `UserMapper.selectPage` 按 username/nickname/role 查询。面试说法：标准分页查询。

### `GET /api/admin/users/{id}` -> `AdminUserController.getUserDetail`

- 作用：查单个用户。Service `loadUser` 查不到抛 404，然后 `AdminUserViewAssembler.toVO`。面试说法：这里不会把 password 暴露出去，VO 里只有基础字段。

### `GET /api/admin/notes` -> `AdminNoteController.getNoteList`

### `PUT /api/admin/wishes/{id}/status` -> `AdminWishController.updateWishStatus`

# 第六部分：核心链路深挖（面试重点）

## 6.1 登录链路

- 请求入口：`POST /api/auth/login`
- 关键类：`AuthController` -> `AuthApplicationService` -> `AuthServiceImpl` -> `AuthRateLimitSupport` / `EmailService` / `TurnstileVerifier` / `JwtUtils`
- 技术参与点：Validation、Redis 限流、邮箱验证码、BCrypt、JWT、Spring Security
- 数据流向：前端提交账号或邮箱 -> DTO `LoginRequest` -> 应用层判断登录方式 -> `UserMapper` 查用户 / Redis 校验验证码 -> 密码校验或验证码校验 -> 生成 token -> 返回 `token + user + avatar + checkin`
- 面试讲法：登录链路我拆成两层，Controller 只接请求，`AuthApplicationService` 负责决定是密码登录还是验证码登录，`AuthServiceImpl` 负责查库、风控、密码校验和 token 生成。

## 6.2 注册链路

- 请求入口：`POST /api/auth/register`
- 关键类：`AuthController.register`、`AuthApplicationService.register`、`EmailService`、`AuthServiceImpl.register`
- 技术参与点：Validation、邮箱验证码、Redis 冷却、BCrypt、JWT、数据库唯一约束、Turnstile
- 数据流向：前端填邮箱/密码 -> 校验验证码 -> 校验密码强度和确认密码 -> Redis 注册频率限制 -> `users` 表 insert -> JWT 返回
- 面试讲法：注册不是单一 insert，而是验证码、风控、密码强度、唯一约束几层一起做。

## 6.3 获取当前用户信息链路

- 请求入口：`GET /api/auth/profile`
- 关键类：`JwtAuthenticationFilter`、`CurrentUserService`、`AuthServiceImpl.getProfile`
- 技术参与点：JWT、Spring Security、Redis 头像
- 数据流向：请求头 token -> 过滤器解析 -> `Authentication` 注入 Controller -> `CurrentUserService.requireUsername` -> `UserMapper` 查库 -> 组装返回
- 面试讲法：当前用户信息不是前端传用户 id，而是 token 解析后由后端自己取。

## 6.4 用户信息更新链路

- 请求入口：`PUT /api/auth/profile`
- 关键类：`AuthController.updateProfile`、`AuthServiceImpl.updateProfile`、`AvatarSupport`
- 技术参与点：JWT、Validation、BCrypt、Redis
- 数据流向：前端提交昵称/密码/头像 -> 当前用户鉴权 -> 改 MySQL 用户表 + 改 Redis 头像 -> 返回最新资料
- 面试讲法：这个接口体现了数据库与 Redis 双写场景。

## 6.5 列表查询 / 分页链路

- 请求入口：如 `GET /api/questions`、`GET /api/notes`、`GET /api/admin/users`
- 关键类：对应 `*ApplicationService.get*List`
- 技术参与点：MyBatis-Plus `Page`、`LambdaQueryWrapper`、分页插件
- 数据流向：Query 参数 -> Service 规范化 page/size -> Wrapper 拼条件 -> `selectPage` -> Entity 转 VO -> `Result.success(page)`
- 面试讲法：分页接口统一走 MyBatis-Plus，Controller 不碰 SQL，Service 负责条件拼接和 VO 组装。

## 6.6 文件上传链路

- 请求入口：`POST /api/question-banks/import`、`POST /api/admin/question-banks/official/import`
- 关键类：`QuestionBankController` / `AdminQuestionBankController`、`QuestionBankImportService`
- 技术参与点：`MultipartFile`、TXT 解码、事务、MySQL
- 数据流向：上传 TXT -> 校验大小 -> 解析文本块为题目草稿 -> 普通用户场景先存 `custom_question_banks` 等待审核；管理员场景直接导入 `questions`
- 面试讲法：项目里的上传不是存 OSS，而是读内存、解析题目结构、再决定是存原始题库还是直接导题。

## 6.7 邮件验证码链路

- 请求入口：`POST /api/auth/email/send-code`
- 关键类：`AuthApplicationService.sendEmailCode`、`EmailService`
- 技术参与点：Redis TTL、Resend 邮件 SDK、邮箱格式校验、IP 冷却
- 数据流向：前端给 email/type -> 校验账号状态 -> Redis 存 6 位验证码和冷却 -> Resend 发邮件 -> 注册/登录/重置时再校验 Redis 中的 code
- 面试讲法：Redis 在这里主要解决验证码有效期和防刷，不只是缓存。

## 6.8 AI 调用链路

- 请求入口：`/api/ai/question/*/explain`、`/api/ai/coach/chat`、`/api/ai/resume/analyze`、`/api/ai/notes/summarize`
- 关键类：`AiController`、`AiServiceImpl`、`AiPromptFactory`、`AiModelGateway`、`AiQuotaUsageSupport`
- 技术参与点：Spring AI、SSE、Redis 会话、Redis 配额、联网搜索/天气补充
- 数据流向：请求 -> 身份识别 -> 配额判断 -> Prompt 生成 -> `ChatClient` 调模型 -> SSE 或普通结果返回 -> Redis 记历史和 token 用量
- 面试讲法：AI 层做成了服务网关，不让 Controller 直接拼 Prompt 或直接打模型。

## 6.9 游客模式链路

- 请求入口：`POST /api/auth/guest`，后续用于排行榜/愿望墙/笔记等受限逻辑判断
- 关键类：`AuthServiceImpl.guestLogin`、`CurrentUserService`、`WishWallController.resolveActor`
- 技术参与点：JWT、角色识别
- 数据流向：游客拿 GUEST token -> 后续请求里 `CurrentUserService` 把它识别成 guest -> 某些接口允许浏览，某些接口比如发愿望/评论会拒绝
- 面试讲法：游客模式本质是发一个 GUEST 角色 token，而不是完全匿名。

## 6.10 权限校验链路

- 请求入口：所有带 token 的接口
- 关键类：`SecurityConfig`、`JwtAuthenticationFilter`、`CurrentUserService`、`QuestionAccessSupport`
- 技术参与点：Spring Security Filter Chain、JWT、角色控制、对象级权限
- 数据流向：过滤器先恢复身份 -> SecurityConfig 先做 URL 级权限 -> Service 再做对象级权限，比如题库 owner 校验、笔记 owner 校验
- 面试讲法：项目里既有 URL 级权限，也有资源级权限；前者靠 Spring Security，后者靠业务 Service。

# 第七部分：项目中的技术点面试化总结（结合真实项目）

## 7.1 你的项目整体架构是什么样的？

- 真实实现：项目是 Spring Boot 单体后端，按 `controller -> applicationService/service -> mapper -> mysql/redis` 分层，同时在 `module/*` 里按业务拆包。
- 面试怎么说：我的项目整体是前后端分离的单体后端，Web 层用 Spring MVC，认证用 Spring Security + JWT，数据层用 MyBatis-Plus 操作 MySQL，Redis 负责验证码、限流、队列、排行榜和 AI 配额，AI 功能单独封成网关。
- 优化方向：统一新旧两套 service 风格，减少混搭。

## 7.2 为什么要分 Controller、Service、Mapper？

- 真实实现：Controller 只接 HTTP；Service 处理业务；Mapper 只做数据访问。
- 面试怎么说：比如登录接口里，Controller 只接 `LoginRequest`，`AuthApplicationService` 负责业务分流，`AuthServiceImpl` 负责认证逻辑，`UserMapper` 只负责查用户，这样每层职责明确。
- 优化方向：进一步把跨模块支撑能力抽成统一 domain/service 组件。

## 7.3 你项目中的登录鉴权是怎么做的？

- 真实实现：登录成功后 `AuthServiceImpl.buildAuthResult` 调 `JwtUtils.generateToken` 发 token，请求进入时 `JwtAuthenticationFilter` 解析 token 并写入 `SecurityContextHolder`。
- 面试怎么说：项目采用无状态 JWT 鉴权，登录时签发 token，后续通过 Security 过滤器恢复登录身份，Controller 直接注入 `Authentication` 使用。
- 优化方向：补真正 logout 和 token 黑名单。

## 7.4 JWT 在你项目里怎么用的？

- 真实实现：存 `username` 和 `role`；Filter 解析后生成 `UsernamePasswordAuthenticationToken`；密码重置时通过 Redis 记录 token 失效时间。
- 面试怎么说：JWT 主要承载用户名和角色，Security 过滤器解析后就能知道当前是匿名、游客、普通用户还是管理员。
- 优化方向：补 refresh token、token 黑名单、设备维度管理。

## 7.5 Redis 在你项目里用来做什么？

- 真实实现：验证码、邮箱冷却、登录限流、token 失效时间、签到 Bitmap、头像、AI 配额、AI 聊天历史、AI 审计、愿望审核队列、排行榜、访客统计。
- 面试怎么说：Redis 在我项目里承担多种高频状态存储，不只是缓存，像验证码、队列、HyperLogLog UV 统计、ZSet 排行榜都在用。
- 优化方向：统一 key 规范和监控告警。

## 7.6 MyBatis-Plus 在你项目里是怎么简化开发的？

- 真实实现：全部 Mapper 继承 `BaseMapper`，列表接口统一 `Page + LambdaQueryWrapper`，更新时用 `updateById` 或 `UpdateWrapper`。
- 面试怎么说：像题目列表、笔记列表、后台用户列表都不用手写 XML，分页和条件查询基本靠 MyBatis-Plus 完成。
- 优化方向：复杂统计可继续拆到专门 SQL 或读模型。

## 7.7 DTO、VO、Entity 为什么要分开？

- 真实实现：`LoginRequest/RegisterRequest/NoteCreateRequest` 接收前端；`User/Question/Note` 对应表；`NoteDetailVO/QuestionAnswerVO/AdminUserVO` 返回前端。
- 面试怎么说：如果直接用 Entity 接前端，很容易把数据库字段暴露出去，也不利于后续字段演进，所以我把请求对象、数据库对象、返回对象分开了。
- 优化方向：把现在仍然返回 `Map<String,Object>` 的接口改成明确 VO。

## 7.8 全局异常处理在你项目里怎么落地的？

- 真实实现：`GlobalExceptionHandler` 统一处理 `BusinessException`、参数校验异常、`IllegalArgumentException`、运行时异常。
- 面试怎么说：业务层可以直接抛 `NotFoundException/ForbiddenException`，Controller 不需要写 try-catch，最后统一转成 `Result.error`。
- 优化方向：加 traceId 和更细的错误码体系。

## 7.9 统一返回结果为什么要做？

- 真实实现：所有 Controller 统一返回 `Result<T>`。
- 面试怎么说：前端只处理 `code/msg/data` 三段，成功失败结构一致，联调更简单。
- 优化方向：增加 timestamp、traceId、path。

## 7.10 分页接口怎么实现？

- 真实实现：Controller 收 `page/size`，Service 用 `Page<T>` 和 `selectPage` 查询，再把 Entity page 转成 VO page。
- 面试怎么说：分页逻辑统一放 Service，不让 Controller 碰 SQL 或分页细节。
- 优化方向：统一最大页大小限制。

## 7.11 如果有 AI 接口，怎么封装调用？

- 真实实现：Controller 不直接调模型，而是 `AiServiceImpl -> AiPromptFactory -> AiModelGateway`。
- 面试怎么说：我把 AI 能力拆成 Prompt 组装、配额控制、会话历史、模型调用网关，避免业务层散落着 Prompt 和 HTTP 请求。
- 优化方向：再补模型切换策略和监控面板。

## 7.12 如果有文件上传，怎么处理？

- 真实实现：当前只支持 TXT 题库上传，通过 `MultipartFile` 读 bytes，`QuestionBankImportService` 解析文本块，不做文件落盘。
- 面试怎么说：上传逻辑本质是“读取内容 -> 解析成题目草稿 -> 持久化结果”，不是 OSS 场景。
- 优化方向：补类型白名单、批量导入、异步解析。

## 7.13 如果有邮件服务，怎么接入？

- 真实实现：`EmailService` 使用 Resend SDK，验证码先落 Redis 再发送邮件。
- 面试怎么说：邮件服务主要做验证码通知，Redis 负责 TTL 和防刷，Resend 负责外部投递。
- 优化方向：做异步发送和投递失败重试。

## 7.14 如果有过滤器，它是怎么参与请求流程的？

- 真实实现：`JwtAuthenticationFilter` 继承 `OncePerRequestFilter`，在 Security 链中先于 `UsernamePasswordAuthenticationFilter` 执行。
- 面试怎么说：过滤器会在 Controller 前解析 Authorization 头，把用户身份塞进 SecurityContext，Controller 再通过 `Authentication` 使用。
- 优化方向：增加日志和异常原因可观测性。

## 7.15 如果有参数校验，是怎么做的？

- 真实实现：DTO 字段上配 `@NotBlank/@Size/@Email/@Min/@Max`，Controller 参数加 `@Valid/@Validated`。
- 面试怎么说：先把明显非法参数挡在 Controller 门口，业务层再处理业务规则校验。
- 优化方向：所有 Map 入参补 DTO。

## 7.16 如果项目中有不合理的地方，你会怎么优化？

- 真实实现：主要问题包括 `logout` 空实现、部分笔记接口权限不严、配置默认值含敏感信息、架构风格混搭。
- 面试怎么说：我会优先补安全和边界问题，比如 token 失效、对象级权限、配置脱敏，然后再统一架构风格和响应对象。
- 优化方向：见下一部分详细排查。

# 第八部分：AI 风格代码与问题代码排查

## 8.1 明显像 AI 直接生成或拼接出来的地方

- `AuthController.logout`：接口存在，但业务为空，典型“先把接口补齐、逻辑后补”的 AI 风格占位实现。
- `AdminDashboardApplicationService.getDashboard`：`aiCallCount` 直接写死 `4520`，很像为了先把页面跑起来临时填的常数。
- 多个类广泛返回 `Map<String,Object>`，如 `AuthServiceImpl.getProfile/buildAuthResult`、`LeaderboardServiceImpl.getLeaderboard`、`AnalyticsServiceImpl.getVisitorRecords`，说明类型设计不够统一，偏“先能跑”。
- 项目里大量 `safe()/trim()/defaultText()/normalizeXxx()` 小工具方法散落在各类中，说明封装粒度有些碎，像多轮生成后逐步打补丁形成的代码风格。

## 8.2 表面能跑，但逻辑不够严谨的地方

- `AuthController.logout`：前端删 token 不等于真正退出，服务端 token 依然有效。
- `NoteApplicationService.getNoteDetail`：没有限制只能看公开笔记或自己的笔记，存在越权看待审核笔记的风险。
- `NoteApplicationService.increaseView`：对任何存在的笔记都能加浏览量，而且是“查出 -> +1 -> updateById”，并发下可能丢增量。
- `NoteApplicationService.toggleLike`：没有校验笔记状态是否公开。
- `NoteApplicationService.getComments/addComment`：没有先校验笔记是否存在且允许评论。
- `AiController` 里有匿名 IP 回退逻辑，但 Security 规则并没有对大多数 AI 接口放开，说明“匿名 AI 能力”代码和权限配置没有完全对齐。

## 8.3 命名不够规范或有风险的地方

- `service` 和 `module/*/service` 并存，`ApplicationService`、`ServiceImpl` 两套命名风格混搭，阅读成本高。
- `application-dev.yml` 中含有 JWT secret 和 AI key 默认值，这不是命名问题，而是配置治理问题。
- 部分中文/emoji 字面量在终端显示为乱码，说明源码编码和终端编码存在不一致风险，尤其 `AvatarSupport` 的头像集合和大量中文提示语。

## 8.4 分层不够合理的地方

- `AuthController.sendEmailCode` 直接收 `Map<String,String>`，没有 DTO，导致参数层不整齐。
- `AnalyticsServiceImpl.getVisitorRecords` 在 Service 里手动做过滤分页拼装，随着数据量变大维护成本会上升。
- `AdminNoteController.updateNoteStatus` 同时兼容 query 和 body 两种输入，接口语义不够干净。

## 8.5 异常处理不足的地方

- `WishServiceImpl` 和 `AnalyticsServiceImpl` 很多地方用了 `catch (Exception ignore)` 或日志后吞掉异常，虽然保证系统不炸，但定位问题不够精确。
- `AiModelGateway` 失败时大量走 fallback，是对用户友好，但如果没有外部监控，很容易掩盖真实错误。

## 8.6 安全性不足的地方

- `application-dev.yml` 和默认配置里出现真实风格 API key / JWT secret，占位值不应进仓库。
- `AuthServiceImpl.updateProfile` 修改密码后没有主动使旧 token 失效。
- `AiIdentitySupport.resolveRole` 通过用户名前缀判断 guest，和 `guest-uuid` 的真实生成规则不完全一致，可能导致游客被当作 USER 计额度。
- `UserUsernameUniqGuard` 启动时自动改线上数据并加索引，适合作为一次性修复脚本，不太适合长期默认放在应用启动流程。

## 8.7 面试时容易被追问的地方

- 为什么 `logout` 不做 token 失效？
- 为什么笔记详情接口没做状态控制？
- 为什么有些接口返回 `Map` 而不是强类型 VO？
- 为什么 Redis 同时承担这么多职责，是否会造成 key 管理混乱？
- 为什么 AI 模块既能联网又能流式，还如何做限流和降级？

## 8.8 企业级项目一般会怎么优化

- 补强类型化响应 VO，减少 `Map<String,Object>`。
- 补真正 logout / refresh token / token 黑名单。
- 把配置默认敏感值全部移出仓库。
- 为笔记、愿望、题库等资源补更严的对象级权限控制。
- 愿望队列改成真正的消息队列或 Redis Stream，提升可观测性和消费语义。
- 批量统计接口增加缓存层、索引优化和监控。

# 第十部分：最终面试答案（可以直接拿去说）

## 10.1 项目整体介绍

> 我这个项目叫 NovaLeap，是一个面向面试学习场景的 Java 后端项目。整体用的是 Spring Boot 3 做单体后端，Web 层是 Spring MVC，登录鉴权用 Spring Security 加 JWT，数据库层用 MyBatis-Plus 操作 MySQL，Redis 主要承担验证码、限流、排行榜、签到、访客分析、AI 配额和异步队列。业务上主要有认证、题库、用户自定义题库、笔记、愿望墙、排行榜、访客分析和 AI 面试教练几个模块。项目结构不是完全传统三层，而是 Controller 下接 ApplicationService，再往下走 Mapper 和一些 Support 组件，AI 和认证这些跨模块能力还保留了传统 service/impl 实现。

## 10.2 技术栈介绍

> 这个项目核心技术栈是 Spring Boot、Spring MVC、Spring Security、JWT、MyBatis-Plus、MySQL、Redis、Spring AI。除了这些，我还用了 Validation 做参数校验，`@RestControllerAdvice` 做全局异常处理，MyBatis-Plus 分页插件做列表分页，Resend 做邮箱验证码发送，JDK HttpClient 做 Turnstile 校验、IP 地理位置解析和 AI 外部上下文补充，另外愿望墙审核用了 Redis List/Set 加定时任务实现异步消费。

## 10.3 登录鉴权介绍

> 登录这块我做了密码登录、邮箱验证码登录和游客登录三种方式。前端调 `/api/auth/login` 时，Controller 先接 `LoginRequest`，应用层判断是密码模式还是验证码模式。密码模式下会查 MySQL 用户表、做 Redis 登录限流、BCrypt 密码校验；验证码模式下会先到 Redis 校验邮箱验证码。登录成功后在 `AuthServiceImpl.buildAuthResult` 里调用 `JwtUtils.generateToken` 生成带用户名和角色的 token。后续请求进入系统时，`JwtAuthenticationFilter` 会从 Authorization 头解析 Bearer token，恢复出 `Authentication` 放到 SecurityContext 里，Controller 或 Service 再通过 `CurrentUserService` 获取当前用户身份。

## 10.4 Redis 介绍

> Redis 在我项目里不是简单做缓存，而是承担了很多状态类职责。认证模块里用它存邮箱验证码、邮箱冷却、IP 冷却、登录失败次数和 token 失效时间；签到模块用 Bitmap 存连续签到；AI 模块用它存聊天历史、调用配额、冷却时间和审计记录；排行榜模块用 ZSet 和 Set 存积分和已做题集合；访客分析模块用 String、Hash、ZSet、HyperLogLog 统计 PV/UV 和访客画像；愿望墙模块则用 List、Set 做待审核队列、处理中队列和死信队列。所以面试时我一般会说，Redis 在项目里承担的是“高频状态存储中心”。

## 10.5 MyBatis-Plus 介绍

> 我项目数据库层主要用 MyBatis-Plus。所有 Mapper 都继承 `BaseMapper`，普通增删改查基本不写 XML。像题目列表、笔记列表、后台用户列表这些分页接口，都是 `Page<T>` 配合 `LambdaQueryWrapper` 做条件查询，再通过分页插件自动生成分页 SQL。更新时常用 `updateById`，复杂一点会用 `UpdateWrapper`，比如题目浏览量增加和笔记审核状态更新。它最大的价值是让我把精力更多放在业务逻辑和权限判断上，而不是重复写 SQL 样板。

## 10.6 讲一个你做过的接口

> 我比较常讲的是“用户上传自定义题库并审核导入”这条链路。普通用户先通过 `/api/question-banks/import` 上传 TXT 文件，后端会校验文件大小，然后由 `QuestionBankImportService` 按题目块解析文本，先把原始内容和元信息存进 `custom_question_banks` 表，状态设成待审核。管理员再调 `/api/admin/question-banks/{id}/audit` 审核，如果通过，`AdminQuestionBankAuditService` 会把 rawContent 再解析成多条 `Question`，写入 `questions` 表，并打上 `sourceType=CUSTOM` 和 `customBankId`。前台查题时，`QuestionAccessSupport` 会判断当前用户是不是这个自定义题库的拥有者，只有有权限的人才能看到这些题。这个接口能体现文件上传、解析、审核流、批量入库和对象级权限几个点。

## 10.7 项目亮点介绍

> 我觉得项目亮点有三个。第一，Redis 用得比较深入，不只是缓存，还做了验证码、限流、签到、排行榜、访客分析、AI 配额和异步队列。第二，AI 能力不是简单调第三方接口，而是拆成了 Prompt 工厂、模型网关、会话历史、配额控制和 SSE 流式输出。第三，愿望墙做了异步审核链路，用户提交后先入库再入 Redis 队列，定时任务消费时再结合 AI 补情绪、颜色和浮动参数，失败还能进死信并由后台重试，这条链路很适合拿来讲工程化设计。
